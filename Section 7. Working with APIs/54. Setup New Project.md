## Introduction

In this lesson, we begin the **API Testing** section by creating a brand-new Playwright project specifically for testing APIs.

Instead of continuing with the previous **Nebular UI application**, we'll use a different application called **Conduit**, which provides both:

- A web interface (Frontend)    
- A REST API (Backend)

This application is ideal for learning API testing because every action performed in the browser communicates with backend APIs, allowing us to inspect, intercept, mock, and directly test API requests.

# Understanding the Application Flow

The Conduit application follows a standard client-server architecture.

```text
Browser (Client)
        │
        │ HTTP Requests
        ▼
Conduit Backend API
        │
        ▼
Database
        │
        ▼
HTTP Responses
        │
        ▼
Browser Updates UI
```

Every interaction in the UI results in one or more API requests.

# Inspecting API Calls

Open the application in Chrome.

Then open:

```text
Developer Tools (F12)
```

Go to:

```text
Network
```

Filter requests by:

```text
Fetch/XHR
```

Now refresh the page.

You'll immediately notice multiple API requests.

---

# Example: Loading Tags

When the homepage loads, the browser sends a request similar to:

```http
GET /api/tags
```

The response might look like:

```json
{
    "tags": [
        "welcome",
        "javascript",
        "playwright",
        "testing"
    ]
}
```

The browser takes this JSON data and renders the list of tags on the right side of the page.

This demonstrates a key concept:

> **The UI simply displays data returned by the API.**

---

# Example: Loading Articles

Another request retrieves articles.

```http
GET /api/articles
```

Response:

```json
{
    "articles": [
        {
            "title": "First Article",
            "description": "...",
            "body": "..."
        }
    ]
}
```

The articles displayed on the homepage are generated entirely from this API response.

# What Happens Behind the Scenes?

When you click **Publish Article**, the browser sends a request like:

```http
POST /api/articles
```

Request Body:

```json
{
    "article": {
        "title": "This is a title",
        "description": "This is a description",
        "body": "This is a test body"
    }
}
```

The API creates the article and returns the newly created article as the response.

Immediately afterward, additional requests are made:

- Get article details    
- Get article comments

Since it's a new article, the comments response is empty.

---

# Deleting an Article

When you click:

```text
Delete Article
```

The browser sends:

```http
DELETE /api/articles/{slug}
```

The backend deletes the article, and the UI updates accordingly.

# Creating a Common Setup

Since every test begins on the application's homepage, use a `beforeEach` hook.

```typescript
import { test, expect } from "@playwright/test";

test.beforeEach(async ({ page }) => {
  await page.goto("https://conduit.bondaracademy.com/");
});
```

Now every test automatically opens the application.

---

# Writing the First Test

Verify that the application logo displays the correct text.

```typescript
test("Application loads successfully", async ({ page }) => {
  await expect(page.locator(".navbar-brand")).toHaveText("conduit");
});
```

If the assertion passes, the new Playwright project has been configured correctly.

# Installing Postman

The instructor also recommends installing **Postman**.

Postman is a popular API client that allows you to:

- Send GET requests    
- Send POST requests
- Test PUT requests
- Test DELETE requests
- Inspect responses
- View request headers
- Experiment with APIs without writing code

Although Playwright can perform API testing, Postman is extremely useful for learning, debugging, and exploring REST APIs.
