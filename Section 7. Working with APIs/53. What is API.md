## Introduction

In this lesson, we begin exploring **API Testing in Playwright**. Before writing API tests, it's important to understand what an **API (Application Programming Interface)** is, how it works, the different types of API requests, HTTP status codes, and how Playwright interacts with APIs.

Modern web applications are built using a **client-server architecture**, where the browser (client) communicates with a backend server through APIs. Playwright can interact with these APIs directly, intercept them, or even mock them to create faster and more reliable automated tests.

---

# What is an API?

**API (Application Programming Interface)** is a communication interface that allows two software applications to exchange data.

Think of an API as a **middleman** between:

- Client (Browser, Mobile App, Desktop App)    
- Server (Backend)

The client sends a request, and the server processes it and returns a response.

### Example

Imagine a Weather API with two endpoints:

```http
GET /weather
GET /forecast
```

Request:

```json
{
    "city": "New York"
}
```

Response:

```json
{
    "temperature": "70°F",
    "humidity": "45%"
}
```

For another endpoint:

```http
GET /forecast
```

Response:

```json
{
    "tomorrow": "72°F",
    "dayAfterTomorrow": "68°F"
}
```

The browser doesn't know how to calculate the weather.

Instead, it asks the API.

The API processes the request and returns the result.

---

# How APIs Work

The overall communication flow looks like this:

```text
Client (Browser)
        │
        │ HTTP Request
        ▼
    API Server
        │
        │ Process Request
        ▼
     Database / Business Logic
        │
        │ Response
        ▼
Client (Browser)
```

The browser simply requests data.

The server performs the work and sends the response.

# Client-Server Architecture

Modern applications follow this architecture.

```text
+------------------+
|    Browser       |
|    (Client)      |
+--------+---------+
         |
         | HTTP Request
         |
         ▼
+------------------+
|      API         |
| (Backend Server) |
+--------+---------+
         |
         ▼
+------------------+
|    Database      |
+------------------+
```

The browser never talks directly to the database.

Everything goes through the API.

# Common HTTP Request Types

There are many HTTP methods, but four are used most frequently.

| Method | Purpose              |
| ------ | -------------------- |
| GET    | Retrieve data        |
| POST   | Create new data      |
| PUT    | Update existing data |
| DELETE | Remove data          |

---

## GET Request

Used to retrieve information.

Example:

```http
GET /users/15
```

Returns:

```json
{
    "id": 15,
    "name": "John"
}
```

No data is modified.

---

## POST Request

Used to create a new resource.

Example:

```http
POST /users
```

Request Body:

```json
{
    "name": "John",
    "email": "john@test.com"
}
```

The server creates a new user.

---

## PUT Request

Updates existing data.

Example:

```http
PUT /users/15
```

Request:

```json
{
    "email": "newemail@test.com"
}
```

The user's email is updated.

---

## DELETE Request

Deletes a resource.

Example:

```http
DELETE /users/15
```

The user is removed from the database.

---

# Anatomy of an API Request

A typical API request contains four main parts.

## 1. URL (Endpoint)

The API address.

Example:

```http
https://example.com/api/users
```

---

## 2. Headers

Provide additional information.

Common headers:

```http
Content-Type: application/json

Authorization: Bearer token123
```

Headers often specify:

- Content type    
- Authentication
- API version
- Language

---

## 3. HTTP Method

Examples:

```text
GET
POST
PUT
DELETE
```

---

## 4. Request Body

Usually included only with:

- POST    
- PUT
- PATCH

Example:

```json
{
    "username": "John",
    "password": "Welcome1"
}
```

---

# API Response

The server sends back:

- Status Code    
- Headers
- Response Body

Example:

```http
200 OK
```

```json
{
    "message": "Success"
}
```

---

# HTTP Status Codes

Status codes tell whether the request succeeded or failed.

---

## 2xx — Success

Everything worked.

Examples:

| Code | Meaning    |
| ---- | ---------- |
| 200  | OK         |
| 201  | Created    |
| 204  | No Content |

Example:

```text
200 OK
```

---

## 3xx — Redirection

The requested resource has moved.

Example:

```text
301 Moved Permanently
302 Found
```

The client is redirected to another URL.

---

## 4xx — Client Errors

The client made an invalid request.

Examples:

| Code | Meaning      |
| ---- | ------------ |
| 400  | Bad Request  |
| 401  | Unauthorized |
| 403  | Forbidden    |
| 404  | Not Found    |

### 400 Bad Request

Incorrect request format.

### 401 Unauthorized

Authentication required.

### 404 Not Found

Requested endpoint doesn't exist.

---

## 5xx — Server Errors

The request was valid, but the server failed.

Examples:

| Code | Meaning               |
| ---- | --------------------- |
| 500  | Internal Server Error |
| 501  | Not Implemented       |
| 502  | Bad Gateway           |

These errors usually indicate problems on the server side.

# API Mocking

Instead of allowing the browser to contact the real server, Playwright can intercept the request and return a fake response.

```text
Browser
    │
    ▼
Playwright
    │
Fake Response
    │
    ▼
Browser
```

Advantages:

- Faster tests
- More stable tests
- Full control over returned data
- No dependency on backend availability

Disadvantages:

- Does **not** test the real backend
- API changes may go unnoticed

API mocking is especially useful when:

- The API is slow.
- The API is unstable.
- The application depends on unreliable third-party services.
- You need deterministic test data.

# API Interception

Playwright can also intercept a real API response before it reaches the browser.

```text
Browser
    │
    ▼
API Server
    │
Real Response
    │
    ▼
Playwright
    │
Modify Response
    │
    ▼
Browser
```

This allows you to:

- Validate responses    
- Modify returned data
- Simulate different backend scenarios

without changing the actual server.

# Direct API Testing

Playwright can communicate directly with APIs without opening a browser.

```text
Playwright
     │
     ▼
API Server
```

This is useful for:

- Creating test data    
- Deleting test data
- Authenticating users
- Backend validation
- Faster test execution

A common use case is **headless authentication**, where Playwright obtains an authentication token via an API instead of logging in through the UI.

Benefits:

- Much faster execution
- Eliminates repetitive UI login steps
- Reduces overall test runtime

# Summary

In this lesson, we learned:

- **API (Application Programming Interface)** enables communication between software applications.    
- Modern applications follow a **client-server architecture**, where the browser communicates with the backend through APIs.
- The four most common HTTP methods are:
    - **GET** → Retrieve data
    - **POST** → Create data
    - **PUT** → Update data
    - **DELETE** → Remove data
- An API request typically includes:
    - URL (Endpoint)
    - Headers
    - HTTP Method
    - Request Body (for POST/PUT/PATCH)
- HTTP status code categories:
    - **2xx** → Success
    - **3xx** → Redirection
    - **4xx** → Client errors
    - **5xx** → Server errors
- Playwright supports several API-related capabilities:
    - Performing regular UI-driven API interactions
    - Mocking API responses for faster and more stable tests
    - Intercepting and modifying API responses
    - Sending direct API requests without opening a browser for tasks like authentication and test data setup

These concepts provide the foundation for the upcoming lessons, where you'll learn how to create API requests, validate responses, mock APIs, intercept network traffic, and integrate API testing into your Playwright automation framework.
