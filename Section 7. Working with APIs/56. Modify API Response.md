This lesson builds on API mocking, but instead of **replacing the response completely**, you now:

> Intercept → Fetch real response → Modify it → Return modified response

This is closer to **real-world testing scenarios** like:

- testing UI behavior with altered backend data    
- simulating edge cases without changing backend
- verifying frontend rendering logic

---

# 🔁 Key Idea

Unlike the previous lesson (mocking), here you **don’t discard the API response**.

You:

```text
Real API Response
        ↓
Playwright intercepts it
        ↓
Modify JSON data
        ↓
Send modified response to browser
```

---

# 🧠 Why this is powerful

You can:

- Change only one article instead of mocking everything    
- Keep backend logic intact
- Test UI rendering flexibility
- Simulate rare backend conditions
- Validate frontend behavior with controlled data

---

# 🌐 Target API

You are intercepting:

```text
/api/articles
```

This endpoint returns:

```json
{
  "articles": [
    {
      "title": "...",
      "description": "...",
      ...
    }
  ]
}
```

You will modify:

- `articles[0].title`    
- `articles[0].description`

---

# ⚙️ Core Flow

```ts
await page.route("*/**/api/articles*", async (route) => {

    // 1. Get real API response
    const response = await route.fetch();

    // 2. Convert to JSON
    const responseBody = await response.json();

    // 3. Modify data
    responseBody.articles[0].title = "This is a test title";
    responseBody.articles[0].description = "This is a description";

    // 4. Return modified response
    await route.fulfill({
        body: JSON.stringify(responseBody),
    });
});
```

# ⚠️ Common Mistake (Important Fix)

Your snippet shows:

```ts
const responseBody = await response.body();
responseBody.articles[0].title = ...
```

❌ This is incorrect because:

- `response.body()` returns a **Buffer**    
- not a JSON object

✔ Correct approach:

```ts
const responseBody = await response.json();
```

# 🔄 Mock vs Modify

| Feature             | Mock API         | Modify API        |
| ------------------- | ---------------- | ----------------- |
| Uses real backend   | ❌                | ✅                 |
| Returns custom data | ✅                | Partially         |
| Keeps backend logic | ❌                | ✅                 |
| Best for            | Isolation        | Realistic testing |
| Risk level          | High abstraction | Low abstraction   |

---

# 💡 When to Use Modify API

Use this pattern when:

- You want real data structure but controlled values    
- You are testing UI rendering logic
- Backend is stable but data is unpredictable
- You want partial overrides (not full replacement)

---

# 🚀 Mental Model

Think of Playwright as sitting between:

```text
Browser ⇄ Playwright ⇄ Backend
```

And for modify mode:

```text
Backend → Playwright (edit) → Browser
```

---

# 🧠 Key Takeaways

- `route.fetch()` lets real API execute    
- `response.json()` gives editable object
- You can modify specific fields only
- `route.fulfill()` sends updated response
- This is **partial mocking / response transformation**
- More realistic than full mocking
