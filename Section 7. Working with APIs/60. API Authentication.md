## Overview

In the previous lesson, authentication was performed through the **UI** by filling the login form, clicking the **Sign In** button, and saving the authenticated browser state to a `user.json` file.

Although this works, UI authentication has some disadvantages:

- It is slower because it launches a browser and performs multiple UI interactions.    
- UI changes (such as modified selectors or redesigned login pages) can break authentication.
- Every test run spends unnecessary time logging in through the interface.

A much better approach is to authenticate **directly through the API**. Since the application's authentication is based on a **JWT access token** stored in **localStorage**, we can obtain that token using an API request and write it directly into the authentication state file.

This makes authentication:

- Much faster
- More reliable
- Independent of UI changes
- Easier to maintain

---

# How Playwright Stores Authentication

Previously, the authentication setup created a file:

```text
.auth/
└── user.json
```

This file stores the browser's authenticated state.

Example:

```json
{
  "cookies": [],
  "origins": [
    {
      "origin": "https://conduit.bondaracademy.com",
      "localStorage": [
        {
          "name": "jwtToken",
          "value": "<JWT_TOKEN>"
        }
      ]
    }
  ]
}
```

Notice:

- No cookies are used.    
- Authentication depends entirely on the JWT token stored inside **localStorage**.

Since we already know how to obtain this token through the API, there is no need to log in through the UI anymore.

# New Authentication Flow (API)

```text
Playwright
    │
    ▼
POST /api/users/login
    │
    ▼
Receive JWT Token
    │
    ▼
Update user.json
    │
    ▼
Authenticated Browser
```

No browser login is required.

---

# Step 1 — Perform Login Through the API

Inside `auth.setup.ts`, use Playwright's `request` fixture to call the login endpoint.

```ts
const response = await request.post(
  "https://conduit-api.bondaracademy.com/api/users/login",
  {
    data: {
      user: {
        email: "email",
        password: "password",
      },
    },
  }
);
```

The request body is identical to what the browser sends when a user logs in through the UI.

---

# Step 2 — Read the Response

The server returns a JSON object.

```ts
const responseBody = await response.json();
```

Example response:

```json
{
  "user": {
    "email": "...",
    "username": "...",
    "bio": null,
    "image": null,
    "token": "eyJhbGc..."
  }
}
```

The JWT token is located at:

```ts
responseBody.user.token
```

Extract it:

```ts
const accessToken = responseBody.user.token;
```

---

# Step 3 — Import the Authentication File

Import the existing authentication state.

```ts
import user from "../.auth/user.json";
```

Now the `user` object contains the parsed JSON from `user.json`.

---

# Step 4 — Replace the Stored Token

Update the JWT token inside the imported object.

```ts
user.origins[0].localStorage[0].value = accessToken;
```

Breaking it down:

```text
origins
   │
   ▼
  [0]
   │
   ▼
localStorage
   │
   ▼
  [0]
   │
   ▼
value
```

After this assignment, the authentication state now contains the new token returned by the API.

---

# Step 5 — Save the Updated File

Import Node.js's filesystem module.

```ts
import fs from "fs";
```

Define the path.

```ts
const authFile = ".auth/user.json";
```

Write the updated object back to disk.

```ts
fs.writeFileSync(
  authFile,
  JSON.stringify(user)
);
```

Now the browser will load with a valid authenticated state.

# Complete API Authentication

```ts
import { test as setup } from "@playwright/test";
import user from "../.auth/user.json";
import fs from "fs";

const authFile = ".auth/user.json";

setup("Authentication", async ({ request }) => {

  const response = await request.post(
    "https://conduit-api.bondaracademy.com/api/users/login",
    {
      data: {
        user: {
          email: "email",
          password: "password",
        },
      },
    }
  );

  const responseBody = await response.json();

  const accessToken = responseBody.user.token;

  user.origins[0].localStorage[0].value = accessToken;

  fs.writeFileSync(authFile, JSON.stringify(user));
});
```

# Eliminating Duplicate Authentication Requests

Previously, every API test repeated these steps:

```ts
POST /users/login
```

								↓

```text
Receive Token
```

								↓

```http
Authorization Header
```

								↓

```text
Actual API Request
```

This duplicated the login request in every test.

---

# Better Solution: Environment Variables

Instead of requesting a token repeatedly, save it once.

```ts
process.env["ACCESS_TOKEN"] = accessToken;
```

Why?

Because Node.js provides global environment variables through:

```ts
process.env
```

Any test can access them.

Example:

```ts
process.env["ACCESS_TOKEN"]
```

This removes the need to log in before every API request.

---

# Updated Authentication Setup

```ts
const accessToken = responseBody.user.token;

user.origins[0].localStorage[0].value = accessToken;

fs.writeFileSync(authFile, JSON.stringify(user));

process.env["ACCESS_TOKEN"] = accessToken;
```

Now the token is available globally.

---

# Configure Global Authorization Header

Instead of manually adding the Authorization header in every request, configure it once in `playwright.config.ts`.

```ts
use: {
  extraHTTPHeaders: {
    Authorization: `Token ${process.env["ACCESS_TOKEN"]}`
  }
}
```

Now every API request automatically includes:

```http
Authorization: Token eyJhbGc...
```

without writing it manually.

Repeated in every test.

### After


```ts

```

API request:

```ts

    data: {...}

```

No authentication code is needed inside the test.

# Advantages of API Authentication

- Faster than UI login because no browser interactions are required.
- More stable since it doesn't depend on page elements or selectors.
- Authentication logic is centralized in one place.
- Eliminates repeated login requests across tests.
- Automatically authenticates all API requests through global headers.
- Produces cleaner, shorter, and more maintainable test code.
- Scales well for large test suites.

---

# Key Takeaways

- UI authentication can be replaced with direct API authentication for improved speed and reliability.
- The application's authentication state is stored in `.auth/user.json`, which contains the JWT token in `localStorage`.
- Playwright's `request.post()` can authenticate directly with the backend and retrieve the access token.
- The JWT token is written into `user.json` using Node.js's `fs.writeFileSync()`, allowing the browser to start in an authenticated state.
- The token is also stored in `process.env["ACCESS_TOKEN"]` for reuse throughout the test suite.
- `extraHTTPHeaders` in `playwright.config.ts` automatically attaches the `Authorization` header to every API request.
- Centralizing authentication eliminates duplicate login logic, reduces maintenance, and results in cleaner, faster, and more scalable Playwright tests.

# Code

```ts
// auth.setup.ts
import { test as setup } from "@playwright/test";
import user from "../.auth/user.json";
import fs from "fs";
  
const authFile = ".auth/user.json";
  
setup("Authentication", async ({ request }) => {
  const response = await request.post(
    "https://conduit-api.bondaracademy.com/api/users/login",
    {
      data: {
        user: { email: "email", password: "password" },
      },
    },
  );
  const responseBody = await response.json();
  const accessToken = responseBody.user.token;
  user.origins[0].localStorage[0].value = accessToken;
  
  fs.writeFileSync(authFile, JSON.stringify(user));
  
  process.env["ACCESS_TOKEN"] = accessToken;
});
```
