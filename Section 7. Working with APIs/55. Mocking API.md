## Overview

**API Mocking** is the process of intercepting an API request made by the browser and returning a custom response instead of allowing the request to reach the real backend server.

Playwright provides the `page.route()` API to intercept network requests and `route.fulfill()` to return mocked responses.

This technique is useful for:

- Speeding up test execution    
- Eliminating dependency on unstable APIs
- Testing edge cases that are difficult to reproduce
- Returning predictable test data
- Running tests even when the backend is unavailable

---

# What Happens Normally?

Without mocking, the browser communicates directly with the backend.

```text
Browser
   │
   │ GET /api/tags
   ▼
Backend API
   │
   │ Returns real tags
   ▼
Browser displays data
```

Example response:

```json
{
    "tags": [
        "welcome",
        "implementations",
        "introduction",
        "javascript",
        "angular"
    ]
}
```

Loading depends entirely on backend speed.

---

# What Happens With Mocking?

Playwright intercepts the request before it reaches the server.

```text
Browser
   │
   │ GET /api/tags
   ▼
Playwright Route
   │
   │ Intercepts request
   │
   ▼
Mock Response
{
   "tags":[
      "automation",
      "playwright"
   ]
}
   │
   ▼
Browser
```

The backend is never contacted.

The browser behaves as if the server returned the mocked data.

---

# Why Mock APIs?

## 1. Faster Tests

Instead of waiting for a remote server:

```text
Browser
	↓

Playwright

	↓

Immediate response
```

No network latency.

---

## 2. Stable Tests

Suppose the real API is:

- Slow    
- Temporarily down
- Returning inconsistent data

Mocking removes these problems.

---

## 3. Predictable Test Data

Instead of receiving hundreds of records:

```json
{
    "tags":[
        "angular",
        "react",
        "vue",
        "javascript",
        "... many more ..."
    ]
}
```

You control exactly what the application receives.

Example:

```json
{
    "tags":[
        "automation",
        "playwright"
    ]
}
```

---

## 4. Test Edge Cases

You can easily simulate:

- Empty responses    
- Invalid responses
- Server errors
- Large datasets

without changing the backend.

# How Playwright Mocks Requests

Playwright uses:

```ts
page.route(urlPattern, handler)
```

Whenever a request matches the URL pattern:

1. Playwright intercepts it.    
2. Executes your callback.
3. Returns whatever response you provide.

---

# Basic Syntax

```ts
await page.route(urlPattern, async (route) => {

    await route.fulfill({
        body: JSON.stringify(data)
    });

});
```

---

# Understanding `page.route()`

```ts
await page.route("*/**/api/tags", async (route) => {

});
```

## URL Pattern

```ts
"*/**/api/tags"
```

This means:

Match **any domain** ending with:

```text
/api/tags
```

Examples:

```http
https://abc.com/api/tags

https://xyz.org/api/tags

https://localhost/api/tags
```

All are intercepted.

---

# Why Use Wildcards?

Instead of:

```http
https://conduit-api.bondaracademy.com/api/tags
```

use

```http
*/**/api/tags
```

Benefits:

- Works across environments
- Works in CI/CD
- Doesn't require changing URLs

---

# Creating Mock Data

Initially the mock data was written directly inside the test.

```ts
const tags = {
    tags: [
        "automation",
        "playwright"
    ]
};
```

Although functional, this approach becomes difficult to maintain for large responses.

---

# Better Practice: Store Mock Data Separately

Project structure:

```text
project
│
├── tests
│      api.spec.ts
│
├── test-data
│      tags.json
```

---

## tags.json

```json
{
    "tags": [
        "automation",
        "playwright"
    ]
}
```

Keeping mock responses in separate files makes tests cleaner and allows reuse across multiple test cases.

---

# Importing JSON

Unlike JavaScript modules:

```ts
import { test } from "@playwright/test";
```

JSON imports use default import syntax:

```ts
import tags from "../test-data/tags.json";
```

No curly braces are required because the entire JSON file is imported as a single object.

---

# Returning the Mock Response

```ts
await route.fulfill({
    body: JSON.stringify(tags)
});
```

---

## Why `JSON.stringify()`?

`route.fulfill()` expects the response body to be a string.

JSON objects must therefore be converted into JSON text.

Example:

Object:

```ts
{
    tags: [
        "automation",
        "playwright"
    ]
}
```

After:

```ts
JSON.stringify(tags)
```

Result:

```json
"{\"tags\":[\"automation\",\"playwright\"]}"
```

This string is sent as the HTTP response body.

---

# Complete Mock Example

```ts
import { test, expect } from "@playwright/test";
import tags from "../test-data/tags.json";

test.beforeEach(async ({ page }) => {

    await page.route("*/**/api/tags", async (route) => {

        await route.fulfill({
            body: JSON.stringify(tags)
        });

    });

    await page.goto("https://conduit.bondaracademy.com/");
});

test("has title", async ({ page }) => {

    await expect(
        page.locator(".navbar-brand")
    ).toHaveText("conduit");

});
```

---

# Why Is `page.route()` Called Before `page.goto()`?

This is one of the most important concepts.

### Correct Order

```text
Configure Route

	↓

Open Website

	↓

Browser Requests API

	↓

Playwright Intercepts

	↓

Mock Returned
```

---

### Incorrect Order

```text
Open Website

	↓

Browser Requests API

	↓

Request Already Sent

	↓

Configure Route

	↓

Too Late
```

The request has already reached the backend.

No mocking occurs.

---

# Request Lifecycle

```text
page.goto()

    │

    ▼

Browser loads page

    │

    ▼

Browser requests:

GET /api/tags

    │

    ▼

Playwright Route?

    │

	Yes ──────────────► route.fulfill()

    │

    ▼

Browser receives mocked JSON

    │

    ▼

Application displays mocked tags
```

# Best Practices

- Configure routes **before** navigation.
- Store mock responses in separate JSON files.
- Use wildcard URL patterns when possible.
- Mock only the APIs necessary for the test.
- Keep mocked responses realistic and maintainable.
- Use API mocking to improve speed and reliability, but periodically run tests against the real backend to catch integration issues.

---

# Key Takeaways

- **API mocking** intercepts browser requests and returns custom responses.
- Use `page.route()` to intercept matching network requests.
- Use `route.fulfill()` to send a mocked HTTP response.
- Always configure routes **before** calling `page.goto()`.
- `JSON.stringify()` converts JavaScript objects into response body strings.
- Store large mock responses in external JSON files for cleaner, reusable tests.
- Wildcard URL patterns (e.g., `"*/**/api/tags"`) make tests more portable across environments.
- API mocking results in **faster, more stable, and deterministic** automated tests while reducing dependency on backend availability.
