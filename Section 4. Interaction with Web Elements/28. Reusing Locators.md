In this lesson, we learn how to **reuse locators** in Playwright to reduce code duplication and make tests cleaner and easier to maintain.

Reusing locators is an important best practice because:

- It improves readability
- Reduces repeated code
- Makes tests easier to update
- Improves maintainability

---

# Base Setup

```typescript
import { test } from "@playwright/test";

test.beforeEach(async ({ page }) => {
  await page.goto("http://localhost:4200/");
  await page.getByText("Forms").click();
  await page.getByText("Form Layouts").click();
});
```

---

# Scenario

We want to automate the following steps for the **Basic Form**:

1. Enter email    
2. Enter password
3. Click checkbox
4. Click submit button
5. Verify email value

---

# 1. Initial Approach (Without Reusing Locators)

```typescript
test("reusing locators", async ({ page }) => {

  await page
    .locator("nb-card")
    .filter({ hasText: "Basic form" })
    .getByRole("textbox", { name: "Email" })
    .fill("test@test.com");

  await page
    .locator("nb-card")
    .filter({ hasText: "Basic form" })
    .getByRole("textbox", { name: "Password" })
    .fill("Welcome123");

  await page
    .locator("nb-card")
    .filter({ hasText: "Basic form" })
    .getByRole("button")
    .click();

});
```

---

# Problem with This Approach

Notice this repeated locator:

```typescript
page.locator("nb-card").filter({ hasText: "Basic form" })
```

It appears multiple times.

This creates:

- Duplication    
- Hard-to-maintain code
- Poor readability

---

# 2. Reusing the Parent Locator

We can store the locator inside a constant.

---

## Refactored Version

```typescript
test("reusing locators", async ({ page }) => {

  const basicForm = page
    .locator("nb-card")
    .filter({ hasText: "Basic form" });

  await basicForm
    .getByRole("textbox", { name: "Email" })
    .fill("test@test.com");

  await basicForm
    .getByRole("textbox", { name: "Password" })
    .fill("Welcome123");

  await basicForm
    .getByRole("button")
    .click();

});
```

---

# Benefits of This Refactoring

Advantages:

- Cleaner code    
- Less duplication
- Easier maintenance
- Better readability

If the locator changes later, we only update it once.

---

# 3. Creating More Specific Reusable Locators

We can increase abstraction further.

---

## Example

```typescript
test("reusing locators", async ({ page }) => {

  const basicForm = page
    .locator("nb-card")
    .filter({ hasText: "Basic form" });

  const emailField = basicForm.getByRole("textbox", {
    name: "Email"
  });

  await emailField.fill("test@test.com");

  await basicForm
    .getByRole("textbox", { name: "Password" })
    .fill("Welcome123");

  await basicForm
    .getByRole("button")
    .click();

});
```

---

# Why This Is Useful

Now:

- `basicForm` represents the form container    
- `emailField` represents a specific field

This creates reusable building blocks inside the test.

---

# 4. Adding More Actions

We can continue using the reusable locator.

---

## Example with Checkbox

```typescript
await basicForm.locator("nb-checkbox").click();
```

This searches for the checkbox only inside `basicForm`.

---

# 5. Adding Assertions

Playwright provides built-in assertions using `expect`.

To use assertions, import `expect`:

---

## Import Statement

```typescript
import { test, expect } from "@playwright/test";
```

---

# Complete Example

```typescript
import { test, expect } from "@playwright/test";

test("reusing locators", async ({ page }) => {

  const basicForm = page
    .locator("nb-card")
    .filter({ hasText: "Basic form" });

  const emailField = basicForm.getByRole("textbox", {
    name: "Email"
  });

  await emailField.fill("test@test.com");

  await basicForm
    .getByRole("textbox", { name: "Password" })
    .fill("Welcome123");

  await basicForm.locator("nb-checkbox").click();

  await basicForm.getByRole("button").click();

  await expect(emailField).toHaveValue("test@test.com");

});
```

---

# Understanding the Assertion

```typescript
await expect(emailField).toHaveValue("test@test.com");
```

### What It Does

- Verifies that the email field contains the expected value    
- Waits automatically until the assertion passes or times out

---

# Why Assertions Are Important

Assertions validate application behavior.

Without assertions:

- Tests only perform actions    
- No verification happens

Assertions ensure:

- Expected results are achieved
- UI behaves correctly

---

# Best Practices for Reusing Locators

## Prefer Reusable Parent Containers

```typescript
const form = page.locator(...)
```

instead of repeating complex locators.

---

## Create Meaningful Variable Names

Good examples:

```typescript
const loginForm
const emailField
const submitButton
```

Avoid vague names like:

```typescript
const a
const element1
```

---

## Keep Locators Close to Usage

Declare locators near where they are used to improve readability.
