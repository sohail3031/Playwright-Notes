In this lesson, we learn how to locate **child elements** in Playwright.

Understanding parent-child relationships in the DOM is very important because many elements are nested inside containers such as:

- cards    
- forms
- tables
- sections
- dialogs

Playwright allows us to locate child elements in multiple ways.

---

# Base Setup

```typescript
import { test } from "@playwright/test";

test.beforeEach(async ({ page }) => {
  await page.goto("http://localhost:4200/");
  await page.getByText("Forms").click();
  await page.getByText("Form Layouts").click();
});
```

---

# Understanding Parent and Child Elements

Suppose we have multiple radio buttons inside an `nb-card`.

Example DOM structure:

```html
<nb-card>
    <nb-radio>Option 1</nb-radio>
    <nb-radio>Option 2</nb-radio>
</nb-card>
```

Here:

- `nb-card` → parent element    
- `nb-radio` → child elements

---

# 1. Finding Child Elements Using Space-Separated Locators

You can locate child elements by separating locators with spaces.

---

## Example

```typescript
test("Locating Child Elements", async ({ page }) => {
  await page
    .locator('nb-card nb-radio :text-is("Option 1")')
    .click();
});
```

---

## How This Works

Playwright searches in this order:

1. Find `nb-card`    
2. Inside it, find `nb-radio`
3. Inside it, find text `"Option 1"`
4. Click the matching element

---

# Rule for Child Locators

When using `locator()`:

```text
parent child grandchild
```

Each space represents a deeper level in the DOM hierarchy.

---

# 2. Chaining Locators

Another way to find child elements is chaining multiple `.locator()` calls.

---

## Example

```typescript
test("Locating Child Elements", async ({ page }) => {

  await page
    .locator("nb-card")
    .locator("nb-radio")
    .locator(':text-is("Option 2")')
    .click();

});
```

---

## Explanation

This performs the same steps:

1. Find `nb-card`    
2. Find child `nb-radio`
3. Find text `"Option 2"`
4. Click the element

---

# Which Approach Is Better?

## Compact Syntax

```typescript
locator("parent child grandchild")
```

### Advantages

- Shorter    
- Cleaner
- Easier to read

---

## Chained Syntax

```typescript
locator("parent").locator("child")
```

### Advantages

- More flexible    
- Easier to debug
- Better for dynamic conditions

Both approaches are valid.

---

# 3. Combining Regular Locators with User-Facing Locators

You can combine:

- `locator()`    
- `getByRole()`
- `getByText()`
- Other user-facing locators

---

## Example

```typescript
test("Locating Child Elements", async ({ page }) => {

  await page
    .locator("nb-card")
    .getByRole("button", { name: "Sign in" })
    .first()
    .click();

});
```

---

## Explanation

Steps:

1. Find all `nb-card` elements    
2. Inside them, find buttons named `"Sign in"`
3. Select the first match
4. Click it

---

# Why Combine Locators?

This helps when:

- The page contains duplicate elements    
- User-facing locators alone are not unique
- You want more precise selection

---

# 4. Finding Elements Using Index (`nth()`)

You can select elements by index using `.nth()`.

---

## Example

```typescript
test("Locating Child Elements", async ({ page }) => {

  await page
    .locator("nb-card")
    .nth(3)
    .getByRole("button")
    .click();

});
```

---

# Understanding `.nth()`

```typescript
.nth(3)
```

means:

- Select the 4th matching element    

Indexes always start from:

```text
0
```

So:

- `0` → first element
- `1` → second element
- `2` → third element
- `3` → fourth element

---

# Related Methods

## First Element

```typescript
.first()
```

## Last Element

```typescript
.last()
```

---

# Why Using Index Is Risky

Using:

- `.nth()`    
- `.first()`
- `.last()`

can make tests fragile.

Why?

- UI order may change
- New elements may be added
- Wrong element may get selected

---

# Recommended Best Practice

Prefer:

- Unique locators    
- User-facing locators
- Parent-child relationships

Avoid relying heavily on:

- Indexes
- Element order
