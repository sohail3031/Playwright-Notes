In this lesson, we learn about **User-Facing Locators** in Playwright.

Playwright strongly recommends using locators that mimic how real users interact with the application instead of relying on implementation details such as:

- CSS classes    
- IDs
- XPath
- Internal HTML structure

This approach follows Playwright’s testing philosophy:

> Test user-visible behavior rather than implementation details.

---

# Why User-Facing Locators?

Imagine we have a **Sign In** button.

A traditional approach might be:

```javascript
page.locator("#signInButton");
```

The problem:

- Users never see the ID
- The test may still pass even if the button text disappears
- The application could be broken from the user's perspective

Instead, Playwright recommends using locators based on:

- Visible text
- Roles
- Labels
- Placeholders
- Titles

This makes tests:

- More readable
- More stable
- More user-centric

---

# Base Setup

```javascript
import { test } from "@playwright/test";

test.beforeEach(async ({ page }) => {
  await page.goto("http://localhost:4200/");
  await page.getByText("Forms").click();
  await page.getByText("Form Layouts").click();
});
```

---

# 1. `getByRole()` — Recommended Locator

The most recommended Playwright locator is:

```javascript
page.getByRole()
```

Roles represent the type of UI element:

- button    
- textbox
- checkbox
- link
- heading
- etc.

---

## Example: Find Textbox by Role

```javascript
test("User Facing Locators", async ({ page }) => {
  await page.getByRole("textbox", { name: "Email" }).first().click();
});
```

### Explanation

- `"textbox"` → role of the element    
- `{ name: "Email" }` → accessible name visible to users
- `.first()` → selects first matching element

---

## Example: Find Button by Role

```javascript
test("User Facing Locators", async ({ page }) => {
  await page.getByRole("button", { name: "Sign in" }).first().click();
});
```

### Why this is good

This closely matches real user behavior:

- User sees a button    
- User sees text `"Sign in"`

---

# Commonly Used Roles

| Role       | Example Element |
| ---------- | --------------- |
| `button`   | Buttons         |
| `textbox`  | Input fields    |
| `checkbox` | Checkboxes      |
| `link`     | Anchor tags     |
| `heading`  | Headings        |

# 2. `getByLabel()`

Used for form elements associated with labels.

---

## Example

```javascript
test("User Facing Locators", async ({ page }) => {
  await page.getByLabel("Email").first().click();
});
```

### Explanation

- Finds the input associated with the label `"Email"`    
- Very useful for forms

---

## Example HTML

```html
<label>Email</label>
<input type="email">
```

---

# 3. `getByPlaceholder()`

Used for elements with placeholder text.

---

## Example

```javascript
test("User Facing Locators", async ({ page }) => {
  await page.getByPlaceholder("Jane Doe").click();
});
```

---

## Example HTML

```html
<input placeholder="Jane Doe">
```

### Explanation

- Uses placeholder text visible to users    
- Very useful for input fields

---

# 4. `getByText()`

Find elements using visible text.

---

## Example

```javascript
test("User Facing Locators", async ({ page }) => {
  await page.getByText("Using the Grid").click();
});
```

### Explanation

- Finds elements containing visible text    
- Useful for buttons, headings, menu items, etc.

---

# 5. `getByTitle()`

Find elements using the HTML `title` attribute.

---

## Example

```javascript
test("User Facing Locators", async ({ page }) => {
  await page.getByTitle("IoT Dashboard").click();
});
```

---

## Example HTML

```html
<a title="IoT Dashboard">Dashboard</a>
```

---

# 6. `getByTestId()`

Used for dedicated testing attributes.

Playwright uses:

```text
data-testid
```

by default.

---

# Adding a Test ID

Example HTML:

```html
<button
  type="submit"
  nbButton
  status="primary"
  data-testid="SignIn"
>
  Sign in
</button>
```

---

# Using `getByTestId()`

```javascript
test("User Facing Locators", async ({ page }) => {
  await page.getByTestId("SignIn").click();
});
```

---

# Why `getByTestId()` Is Useful

Advantages:

- Stable    
- Independent of UI text changes
- Good for dynamic applications

Disadvantage:

- Not fully user-facing
- Users cannot see test IDs

Still considered a good practice in many automation projects.

---

# Locator Priority (Best Practice)

Recommended order:

1. `getByRole()`    
2. `getByLabel()`
3. `getByPlaceholder()`
4. `getByText()`
5. `getByTitle()`
6. `getByTestId()`

---

# Why Playwright Recommends User-Facing Locators

Benefits:

- More resilient tests    
- Better readability
- Tests behave like real users
- Easier maintenance
- Better accessibility support
