In this lesson, we learn about **Assertions** in Playwright.

Assertions are used to verify that the application behaves as expected.

Examples:

- Verify text    
- Verify values
- Verify visibility
- Verify element state
- Verify page behavior

Playwright provides two major types of assertions:

1. **General Assertions**
2. **Locator Assertions**

We will also learn about:

- Soft assertions
- Assertion waiting behavior
- Common mistakes

---

# Base Setup

```javascript
import { test, expect } from "@playwright/test";

test.beforeEach(async ({ page }) => {
  await page.goto("http://localhost:4200/");
  await page.getByText("Forms").click();
  await page.getByText("Form Layouts").click();
});
```

---

# 1. General Assertions

General assertions compare regular JavaScript values.

---

## Example

```javascript
test("assertions", async ({ page }) => {

  // general assertions

  const value = 5;

  expect(value).toEqual(5);

});
```

---

# Understanding the Assertion

```javascript
expect(value).toEqual(5);
```

### What Happens

- Left side → actual value    
- Right side → expected value

Playwright compares both values.

Since:

- actual = `5`
- expected = `5`

The test passes.

---

# Example of a Failing Assertion

```javascript
test("assertions", async ({ page }) => {

  // general assertions

  const value = 5;

  expect(value).toEqual(6);

});
```

---

# Why This Test Fails

Expected:

```text
6
```

Actual:

```text
5
```

Since the values are different, the assertion fails.

---

# Common General Assertion Methods

Playwright provides many assertion methods.

Examples:

| Assertion           | Purpose                           |
| ------------------- | --------------------------------- |
| `toEqual()`         | Exact equality                    |
| `toContain()`       | Check array/string contains value |
| `toBeTruthy()`      | Verify value is truthy            |
| `toBeFalsy()`       | Verify value is falsy             |
| `toBeNull()`        | Verify null                       |
| `toHaveLength()`    | Verify length                     |
| `toBeGreaterThan()` | Numeric comparison                |

---

# 2. General Assertion with Web Elements

We can also use general assertions with values extracted from web elements.

---

## Example

```javascript
test("assertions", async ({ page }) => {

  // general assertions

  const basicForm = page
    .locator("nb-card")
    .filter({ hasText: "Basic form" })
    .locator("button");

  const text = await basicForm.textContent();

  expect(text).toEqual("Submit");

});
```

---

# Important Concept

Notice this line:

```javascript
const text = await basicForm.textContent();
```

`textContent()` returns a Promise.

That is why we MUST use:

```javascript
await
```

Without `await`, the assertion will fail because Playwright receives a Promise instead of actual text.

---

# Common Beginner Mistake

Incorrect:

```javascript
const text = basicForm.textContent();
```

Correct:

```javascript
const text = await basicForm.textContent();
```

Always check whether a method returns a Promise.

---

# 3. Locator Assertions

Locator assertions are smarter assertions designed specifically for web elements.

Instead of extracting the value manually, Playwright handles everything automatically.

---

# Example

```javascript
test("assertions", async ({ page }) => {

  // locator assertions

  const basicForm = page
    .locator("nb-card")
    .filter({ hasText: "Basic form" })
    .locator("button");

  await expect(basicForm).toHaveText("Submit");

});
```

---

# Why Locator Assertions Are Better

This assertion:

```javascript
await expect(basicForm).toHaveText("Submit");
```

automatically:

- Finds the element    
- Waits for the element
- Waits for the text
- Performs the validation

No manual extraction is required.

---

# Important Difference

## General Assertions

```javascript
expect(value).toEqual(...)
```

- No waiting    
- Immediate execution

---

## Locator Assertions

```javascript
await expect(locator).toHaveText(...)
```

- Automatically waits    
- Retries for up to 5 seconds by default
- More reliable for UI testing

---

# Why `await` Is Required for Locator Assertions

Locator assertions are asynchronous.

This is why we use:

```javascript
await expect(locator).toHaveText(...)
```

---

# Common Locator Assertion Methods

| Assertion           | Purpose                        |
| ------------------- | ------------------------------ |
| `toHaveText()`      | Verify text                    |
| `toHaveValue()`     | Verify input value             |
| `toBeVisible()`     | Verify visibility              |
| `toBeHidden()`      | Verify hidden state            |
| `toBeEnabled()`     | Verify enabled                 |
| `toBeDisabled()`    | Verify disabled                |
| `toBeChecked()`     | Verify checkbox/radio selected |
| `toHaveAttribute()` | Verify attribute               |

---

# 4. Soft Assertions

Soft assertions allow the test to continue even if the assertion fails.

---

# Example

```javascript
test("assertions", async ({ page }) => {

  // locator assertions

  const basicForm = page
    .locator("nb-card")
    .filter({ hasText: "Basic form" })
    .locator("button");

  await expect.soft(basicForm).toHaveText("Submit");

});
```

---

# Understanding Soft Assertions

Regular assertion:

```javascript
await expect(...)
```

If it fails:

- Test stops immediately    

---

Soft assertion:

```javascript
await expect.soft(...)
```

If it fails:

- Test continues execution    

---

# Example with Continued Execution

```javascript
test("assertions", async ({ page }) => {

  // locator assertions

  const basicForm = page
    .locator("nb-card")
    .filter({ hasText: "Basic form" })
    .locator("button");

  await expect.soft(basicForm).toHaveText("Submit");

  await basicForm.click();

});
```

---

# Example of a Failing Soft Assertion

```javascript
test("assertions", async ({ page }) => {

  // locator assertions

  const basicForm = page
    .locator("nb-card")
    .filter({ hasText: "Basic form" })
    .locator("button");

  await expect.soft(basicForm).toHaveText("Submit1");

  await basicForm.click();

});
```

---

# What Happens Here

The assertion fails because:

- Actual text = `"Submit"`    
- Expected text = `"Submit1"`

BUT:

- The test still continues
- The button click still executes

---

# Important Correction

The original code had a typo:

Incorrect:

```javascript
await baseForm.click();
```

Correct:

```javascript
await basicForm.click();
```

---

# When to Use Soft Assertions

Soft assertions can be useful when:

- You want multiple validations in a single run    
- You want to collect all failures before stopping

---

# Why Soft Assertions Are Usually Avoided

They may:

- Hide failures    
- Make debugging harder
- Allow tests to continue in unstable states

Use them carefully.

---

# Assertion Timeout Behavior

Locator assertions automatically wait.

Default timeout:

- 5 seconds    

Example:

```javascript
await expect(locator).toHaveText("Submit");
```

Playwright keeps retrying until:

- Assertion passes  
    OR
- Timeout is reached    

---

# General vs Locator Assertions

| Feature                      | General Assertions | Locator Assertions |
| ---------------------------- | ------------------ | ------------------ |
| Works with JS values         | Yes                | No                 |
| Works with locators          | Indirectly         | Yes                |
| Auto waiting                 | No                 | Yes                |
| Requires manual extraction   | Usually            | No                 |
| More suitable for UI testing | Less               | Yes                |

---

# Best Practices

## Prefer Locator Assertions for UI Testing

Better:

```javascript
await expect(locator).toHaveText("Submit");
```

Instead of:

```javascript
const text = await locator.textContent();
expect(text).toEqual("Submit");
```

---

## Always Use `await` with Locator Assertions

Correct:

```javascript
await expect(locator).toHaveText(...)
```

---

## Be Careful with Soft Assertions

Use them only when necessary.

---

## Check for Missing `await`

One of the most common beginner mistakes.
