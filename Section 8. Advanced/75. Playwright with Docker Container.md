# Overview

Docker is a containerization platform that allows applications and their dependencies to run in isolated, portable environments called **containers**. Instead of configuring Playwright separately on every machine, Docker packages everything required—including Node.js, Playwright, browsers, operating system libraries, and project dependencies—into a single container.

For Playwright, Docker offers several advantages:

- Consistent execution across development, testing, and production environments.    
- Eliminates "works on my machine" issues.
- Simplifies CI/CD pipeline setup.
- No manual browser installation.
- Easy environment cleanup.
- Reproducible test execution.

Playwright officially provides Docker images that already include Chromium, Firefox, WebKit, and all required dependencies.

---

# Why Use Docker with Playwright?

Without Docker:

```text
Developer A
├── Node.js v20
├── Chrome v138
├── Ubuntu
└── Tests pass

Developer B
├── Node.js v18
├── Chrome v137
├── Windows
└── Tests fail
```

Different operating systems, browser versions, and installed libraries can cause inconsistent results.

With Docker:

```text
Docker Container
├── Same OS
├── Same Node.js
├── Same Browsers
├── Same Libraries
└── Same Results Everywhere
```

Everyone runs exactly the same environment.

# Step 1: Create a Dockerfile

Create a file named:

```text
Dockerfile
```

Example:

```dockerfile
FROM mcr.microsoft.com/playwright:v1.54.0-jammy

WORKDIR /app

COPY package*.json ./

RUN npm install

COPY . .

CMD ["npx", "playwright", "test"]
```

Explanation:

### Base Image

```dockerfile
FROM mcr.microsoft.com/playwright:v1.54.0-jammy
```

Uses Microsoft's official Playwright image.

---

### Working Directory

```dockerfile
WORKDIR /app
```

All commands execute inside `/app`.

---

### Copy Package Files

```dockerfile
COPY package*.json ./
```

Copies:

- package.json    
- package-lock.json

Docker caches this layer for faster rebuilds.

---

### Install Dependencies

```dockerfile
RUN npm install
```

Installs project dependencies.

---

### Copy Project

```dockerfile
COPY . .
```

Copies the entire project into the container.

---

### Default Command

```dockerfile
CMD ["npx", "playwright", "test"]
```

Runs all Playwright tests when the container starts.

---

# Step 2: Create a .dockerignore File

This prevents unnecessary files from being copied.

Example:

```text
node_modules
playwright-report
test-results
.git
.vscode
```

Benefits:

- Smaller image size    
- Faster builds
- Cleaner containers

---

# Step 3: Build the Docker Image

```bash
docker build -t playwright-tests .
```

Explanation:

```text
docker build
```

Builds the image.

```text
-t
```

Assigns a tag (name).

```text
playwright-tests
```

Image name.

```text
.
```

Uses the current directory.

---

# Step 4: Run the Container

```bash
docker run playwright-tests
```

Docker:

- Starts a container.    
- Executes `npx playwright test`.
- Runs all tests.
- Stops after completion.

---

# Running a Specific Project

```bash
docker run playwright-tests npx playwright test --project=chromium
```

Or:

```bash
docker run playwright-tests npx playwright test tests/login.spec.ts
```

---

# Mounting the Current Project

Instead of rebuilding every time, mount the project into the container:

```bash
docker run \
-v ${PWD}:/app \
playwright-tests
```

Linux/macOS:

```bash
-v $(pwd):/app
```

Windows PowerShell:

```powershell
-v ${PWD}:/app
```

Benefits:

- Immediate code changes.    
- No rebuild required.
- Faster development.

---

# Running in Headless Mode

Containers typically run without a graphical interface.

Playwright automatically runs headless:

```typescript
use: {
    headless: true
}
```

or

```bash
npx playwright test --headed=false
```

---

# Saving Test Reports Outside the Container

Mount a local directory:

```bash
docker run \
-v ${PWD}/playwright-report:/app/playwright-report \
playwright-tests
```

Generated reports remain on your local machine.

---

# Saving Test Results

Similarly:

```bash
docker run \
-v ${PWD}/test-results:/app/test-results \
playwright-tests
```

This preserves:

- Screenshots    
- Videos
- Traces
- Logs

---

# Environment Variables

Pass environment variables into the container:

```bash
docker run \
-e ENV=staging \
playwright-tests
```

Access in Playwright:

```typescript
process.env.ENV
```

Example:

```typescript
baseURL:
process.env.ENV === "staging"
    ? "https://staging.example.com"
    : "https://production.example.com"
```

# Installing Additional Packages

If your project requires extra dependencies:


or include them in `package.json` so `npm install` installs everything automatically.

---

# Advantages of Docker for Playwright

- Consistent environments across machines.    
- No browser installation issues.
- Easier onboarding for new developers.
- Simplified CI/CD integration.
- Portable and reproducible test execution.
- Easy cleanup by removing containers.

---

# Limitations

- Requires Docker to be installed.    
- Initial image download can be large.
- Build time increases after dependency changes.
- GUI debugging is less straightforward (tests generally run headless).

---

# Best Practices

- Use the official Playwright Docker image.    
- Keep the image updated with the Playwright version used in your project.
- Use a `.dockerignore` file to reduce image size.
- Mount reports and test results to the host machine.
- Pass environment variables instead of hardcoding configuration.
- Run tests in headless mode inside containers.
- Include all project dependencies in `package.json`.
