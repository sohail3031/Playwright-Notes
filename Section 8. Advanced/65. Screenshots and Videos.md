## Overview

During test automation, there are situations where you need **visual evidence** of what happened during a test run. Playwright provides built-in support for capturing:

- **Screenshots** – static images of the entire page or specific elements.    
- **Videos** – recordings of the browser session.
- **Trace Viewer** – an interactive debugging tool (generally preferred over screenshots and videos).

Although screenshots and videos are useful, **Playwright Trace Viewer is the recommended debugging tool** because it includes screenshots, DOM snapshots, network requests, console logs, actions, and execution timeline.

---

## When to Use Screenshots

Screenshots are useful for:

- Debugging failed tests    
- Capturing application state
- Sharing UI with teammates
- Recording visual checkpoints
- Attaching evidence to test reports
- Visual documentation

Example:

```text
Fill Login Form
        │
        ▼
Take Screenshot
        │
        ▼
Continue Test
```

---

# Types of Screenshots

Playwright supports three kinds of screenshots:

1. Full Page Screenshot    
2. Element Screenshot
3. Screenshot Buffer (Binary)

---

# 1. Capture a Full Page Screenshot

The simplest way is using:

```ts
await page.screenshot();
```

Normally you'll specify where to save it.

```ts
await page.screenshot({
    path: "screenshots/form-layout-page.png"
});
```

Example:

```ts
await page
    .getByRole("textbox", { name: "Email" })
    .fill("test@test.com");

await page.screenshot({
    path: "screenshots/login-page.png"
});
```

Result:

```text
project/
│
├── screenshots/
│      login-page.png
```

The screenshot is captured exactly at that point in the test.

---

# Full Page vs Viewport Screenshot

Default:

```ts
await page.screenshot();
```

Only captures the **current viewport**.

To capture the entire page:

```ts
await page.screenshot({
    path: "full-page.png",
    fullPage: true
});
```

Example:

```text
Without fullPage

──────────────
Visible Area
──────────────

Only this portion is captured.
```

```text
With fullPage

Entire page

  ↓

Header

Content

Footer

Everything is captured.
```

---

# 2. Capture an Element Screenshot

Sometimes you don't need the whole page.

Example:

Capture only the login form.

Instead of:

```text
Entire Browser Window
```

Capture:

```text
Login Form Only
```

Playwright allows screenshots directly from a locator.

```ts
await page
    .locator(".login-form")
    .screenshot({
        path: "screenshots/login-form.png"
    });
```

Example:

```ts
await page
    .locator(".inline-form")
    .screenshot({
        path: "screenshots/inline-form.png"
    });
```

Only that specific element appears in the image.

This is extremely useful for:

- Forms
- Cards
- Dialogs
- Popups
- Tables
- Charts

# 3. Save Screenshot as a Buffer

Instead of saving the screenshot to disk, Playwright can return it as a binary buffer.

```ts
const buffer = await page.screenshot();
```

The buffer can be:

- uploaded to cloud storage
- sent to Slack
- attached to emails
- stored in a database
- processed by another API

Example:

```ts
const buffer = await page.screenshot();

console.log(buffer);
```

---

## Convert Buffer to Base64

```ts
const buffer = await page.screenshot();

console.log(buffer.toString("base64"));
```

Produces a long Base64 string.

Example:

```text
iVBORw0KGgoAAAANSUhEUgAA...
```

Useful for integrations with:

- Slack
- Microsoft Teams
- REST APIs
- Email services

# Screenshot Options

Example:

```ts
await page.screenshot({

    path: "image.png",

    fullPage: true,

    type: "png"

});
```

Common options:

|Option|Description|
|---|---|
|`path`|File location|
|`fullPage`|Capture entire page|
|`type`|png/jpeg|
|`quality`|JPEG quality|
|`omitBackground`|Transparent background (PNG)|

# Videos

Besides screenshots, Playwright can record videos of browser sessions.

Videos are useful for:

- debugging animations
- demonstrating failures
- sharing execution with teammates
- CI/CD artifacts

---

# Enable Video Recording

Configure inside **playwright.config.ts**

Default:

```ts
use: {

    video: "off"

}
```

Enable videos:

```ts
use: {

    video: "on"

}
```

Every test records a video.

---

# Video Modes

Playwright supports multiple recording strategies.

### Always Off

```ts
video: "off"
```

No videos recorded.

---

### Always On

```ts
video: "on"
```

Every test records a video.

---

### On First Retry

```ts
video: "on-first-retry"
```

Only records if the test fails and is retried.

Very useful for CI pipelines.

---

### Retain on Failure

```ts
video: "retain-on-failure"
```

Videos are recorded for all tests, but only failed test videos are preserved.

This is one of the most commonly used options in production because it saves disk space while retaining useful debugging artifacts.

---

# Default Video Resolution

By default:

```text
800 × 800
```

---

# Configure Video Size

```ts
use: {

    video: {

        mode: "on",

        size: {

            width: 1920,

            height: 1080

        }

    }

}
```

Now videos are recorded in Full HD.

---

# Running with Videos

When videos are enabled, run tests using the Playwright CLI:

```bash
npx playwright test
```

or

```bash
npm run test
```

> **Note:** Video recording may not work when launching tests directly using the VS Code Playwright extension's Run button, depending on the execution method. Running through the Playwright CLI reliably generates video artifacts.

---

# Video Output

Videos are automatically stored under:

```text
test-results/
│
├── test-name/
│      video.webm
```

Example:

```text
test-results/

Login Test/

video.webm
```

# Viewing Videos

Videos can be opened directly.

Or viewed inside the HTML report.

Generate report:

```bash
npx playwright show-report
```

Open a failed test.

You'll see:

- Video
- Trace
- Screenshots
- Logs
- Attachments

Everything in one place.

# Screenshots vs Videos vs Trace Viewer

|Feature|Screenshot|Video|Trace Viewer|
|---|---|---|---|
|Static image|✅|❌|✅|
|Shows animations|❌|✅|Partial (step-by-step snapshots)|
|Step-by-step execution|❌|❌|✅|
|Network requests|❌|❌|✅|
|Console logs|❌|❌|✅|
|DOM snapshots|❌|❌|✅|
|Best for debugging|❌|❌|✅|

# Best Practices

### Use Trace Viewer for debugging

Trace Viewer provides significantly more information than screenshots or videos and should be your primary debugging tool.

---

### Use screenshots as checkpoints

Example:

```text
Login

  ↓

Take Screenshot

  ↓

Submit Form

  ↓

Take Screenshot

  ↓

Logout
```

This helps verify important UI states.

---

### Capture only necessary elements

Instead of capturing an entire page:

```ts
await page.screenshot();
```

Prefer:

```ts
await page.locator(".invoice").screenshot();
```

Smaller images are easier to inspect and consume less storage.

---

### Avoid excessive screenshots

Capturing a screenshot after every action can:

- increase execution time
- consume significant disk space
- make reports harder to navigate

Capture screenshots only at meaningful checkpoints or on failures.

---

### Prefer `retain-on-failure` for videos

Recording videos for every successful test wastes storage.

A practical production configuration is:

```ts
video: "retain-on-failure"
```

This keeps artifacts only when they are actually useful.

---

# Key Takeaways

- Playwright supports **full-page**, **viewport**, and **element-specific** screenshots.
- Use `page.screenshot()` for page-level captures and `locator.screenshot()` for individual UI elements.
- Screenshots can be saved directly to disk or returned as a **binary buffer** for integrations.
- Configure video recording in `playwright.config.ts` using modes such as `"on"`, `"on-first-retry"`, or `"retain-on-failure"`.
- Videos are automatically saved under the `test-results/` directory and are linked in the HTML report.
- **Trace Viewer** remains the preferred debugging tool because it combines visual snapshots with network activity, console logs, DOM state, and execution steps.
- In production automation frameworks, a common practice is to use:
    - **Trace:** `on-first-retry`
    - **Video:** `retain-on-failure`
    - **Screenshots:** `only-on-failure` (or capture manually at important checkpoints)
