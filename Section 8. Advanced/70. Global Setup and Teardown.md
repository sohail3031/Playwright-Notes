# Global Setup vs Project Setup

| Feature        | Global Setup                 | Project Setup                   |
| -------------- | ---------------------------- | ------------------------------- |
| Runs           | Once for the entire test run | Once per project                |
| Scope          | All projects                 | Specific project only           |
| Configuration  | `globalSetup`                | `dependencies`                  |
| Implementation | Exported async function      | Playwright test (`*.setup.ts`)  |
| Typical use    | Framework initialization     | Project-specific initialization |

---

# Why Doesn't Global Setup Use Fixtures?

Fixtures exist only during test execution.

Global Setup runs **before any test starts**, so there are no Playwright fixtures available yet.

That is why you create your own API context:

```ts
const context =
await request.newContext();
```

# Real Enterprise Examples

Global Setup commonly performs tasks such as:

- Authenticating with an identity provider    
- Generating shared API tokens
- Starting mock servers
- Seeding shared reference data
- Running database migrations
- Creating temporary databases
- Starting Docker containers
- Launching local services
- Initializing cloud resources
- Configuring feature flags
- Starting VPN or proxy connections
- Preparing shared test environments

---

# When Should You Use Global Setup?

Use Global Setup when **every project** needs the same preparation.

Examples:

- Login required for all tests    
- Shared environment initialization
- Common test data
- Framework-wide services

---

# When Should You NOT Use It?

Avoid Global Setup for:

- Browser-specific initialization    
- Mobile-only preparation
- API-only seed data
- Tests that require different datasets
- Project-specific resources

Those are better handled with **Project Setup**.

---

# Best Practices

1. Keep Global Setup fast and focused.    
2. Avoid creating unnecessary data.
3. Always clean up shared resources in Global Teardown.
4. Don't put assertions or test logic in Global Setup—it prepares the environment, it doesn't validate behavior.
5. Use Project Setup when only a subset of projects needs special initialization.
6. If using environment variables (e.g., `process.env.ACCESS_TOKEN`), ensure they're set before tests that depend on them.

# Interview Questions

### Q1. What is Global Setup in Playwright?

**Answer:** Global Setup is an exported asynchronous function configured in `playwright.config.ts` that runs once before any tests or projects execute. It prepares the shared test environment.

---

### Q2. What is the difference between Global Setup and Project Setup?

**Answer:** Global Setup runs once for the entire Playwright run and applies to every project. Project Setup runs only for projects that explicitly declare it as a dependency.

---

### Q3. Why can't you use Playwright fixtures inside Global Setup?

**Answer:** Fixtures are created during test execution. Global Setup runs before any tests start, so fixtures such as `page` or `request` are unavailable. If API access is needed, create an `APIRequestContext` manually with `request.newContext()`.

---

### Q4. Why do you need `export default` in Global Setup?

**Answer:** Playwright loads and executes the exported function referenced by `globalSetup` (or `globalTeardown`) in the configuration. Without exporting it, Playwright has nothing to invoke.

---

### Q5. When would you choose Global Setup instead of Project Setup?

**Answer:** Choose Global Setup when the initialization is required by **every** project in the test run, such as obtaining a shared authentication token, starting common infrastructure, or preparing a shared environment. Use Project Setup for initialization needed only by specific projects.
