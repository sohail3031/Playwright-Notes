# What is Mobile Device Emulation?

Mobile device emulation allows Playwright to simulate how your web application behaves on real mobile devices without requiring physical hardware.

Playwright can emulate:

- Screen size (viewport)    
- Touch interactions
- Mobile user agent
- Device scale factor (retina displays)
- Orientation (portrait/landscape)
- Browser behavior
- Mobile rendering

This is especially useful for testing **responsive web applications**.

---

# Why Mobile Testing is Important

Today, a large percentage of users access websites from mobile devices. A feature that works perfectly on a desktop may fail completely on a phone because of:

- Responsive layout changes    
- Hidden navigation menus
- Touch interactions
- Different screen sizes
- Different browser rendering

Example:

Desktop:

```text
----------------------------------------
 Logo  Home  Products  Contact  Login
----------------------------------------
```

Mobile:

```text
--------
 Logo ☰
--------

☰
↓
Home
Products
Contact
Login
```

If your Playwright test assumes the desktop navigation is always visible, it will fail on mobile.

# Creating a Mobile Project

The recommended approach is to create a separate project in `playwright.config.ts`.

```ts
projects: [
  {
    name: "mobile",

    testMatch: "test-mobile.spec.ts",

    use: {
      ...devices["iPhone 13 Pro"],
    },
  },
];
```

This tells Playwright:

- Run only `test-mobile.spec.ts`    
- Use iPhone 13 Pro emulation

# What Does `devices[]` Configure?

When using

```ts
...devices["iPhone 13 Pro"]
```

Playwright automatically configures:

```ts
viewport

userAgent

deviceScaleFactor

isMobile

hasTouch

defaultBrowserType
```

You don't need to configure these manually.

---

# Example

```ts
use: {
    ...devices["iPhone 13 Pro"],
}
```

Much easier than

```ts
use: {

    viewport: {
        width:390,
        height:844
    },

    isMobile:true,

    hasTouch:true,

    deviceScaleFactor:3,

    userAgent:"..."
}
```

---

# Using Viewport Only

Instead of a predefined device, you can specify only a viewport.

```ts
use: {

    viewport:{

        width:390,
        height:844

    }

}
```

This changes only the browser size.

It does **not** emulate:

- Mobile browser    
- Touch events
- User agent

Therefore, using `devices[]` is generally preferred.

# Problem with Responsive Applications

Suppose the desktop layout is

```text
Home
Products
Contact
```

Your test does

```ts
await page.getByText("Products").click();
```

Works on desktop.

---

On mobile

```text
☰
```

The menu is hidden.

The same test fails.

---

# Fixing Mobile Navigation

Click the hamburger menu first.

```ts
await page.locator(".sidebar-toggle").click();
```

Then continue.

```ts
await page.getByText("Forms").click();
```

---

Sometimes the sidebar needs another click to fully close or reveal the page content.

Example:

```ts
await page.locator(".sidebar-toggle").click();

await page.getByText("Forms").click();

await page.locator(".sidebar-toggle").click();
```

# Reusing the Same Test for Desktop and Mobile

Instead of duplicating tests, detect which project is running.

Playwright provides `testInfo`.

Example:

```ts
test("Forms", async ({ page }, testInfo) => {

    if (testInfo.project.name === "mobile") {

        await page.locator(".sidebar-toggle").click();

    }

    await page.getByText("Forms").click();

});
```

Desktop:

```text
Skip mobile-only step
```

Mobile:

```text
Click hamburger menu
```

Same test.

# Running the Same Test on Multiple Devices

You can create multiple projects.

```ts
projects:[

	{
	
		name:"iphone",
		
		use:{
			...devices["iPhone 13"]
		}
	
	},
	
	{
	
		name:"ipad",
		
		use:{
			...devices["iPad Pro"]
		}
	
	},
	
	{
	
		name:"pixel",
		
		use:{
			...devices["Pixel 7"]
		}
	
	}

]
```

Running

```bash
npx playwright test
```

executes the same tests on all configured devices.

# Best Practice: Reuse Tests

Instead of writing

```ts
desktop-login.spec.ts

mobile-login.spec.ts
```

write

```ts
login.spec.ts
```

and adapt only where behavior differs.

Good:

```ts
if(testInfo.project.name==="mobile"){

    await page.locator(".sidebar-toggle").click();

}
```

Avoid duplicating entire test suites.

---

# When Separate Mobile Tests Are Justified

Separate mobile-specific tests make sense when:

- The mobile UI is significantly different from the desktop UI.    
- Mobile-only features exist (e.g., swipe gestures, pull-to-refresh).
- The mobile application (or mobile web version) has a different user flow.

Otherwise, prefer a single reusable test with small conditional logic.

---

# Advantages of Device Emulation

- No physical device required.    
- Fast execution.
- Supports many popular devices.
- Easy responsive testing.
- Runs in CI/CD.
- Same test across multiple screen sizes.
- Verifies layout and navigation changes.

---

# Limitations

Device emulation is powerful but not identical to testing on a real phone.

It does **not** perfectly reproduce:

- Device hardware performance    
- Battery constraints
- OS-level quirks
- Native browser bugs
- Network carrier behavior
- Device sensors (unless explicitly mocked)

For release validation, many teams complement Playwright emulation with testing on real devices or cloud device farms.

---

# Best Practices

- Create separate Playwright projects for desktop, tablet, and mobile.    
- Prefer predefined `devices[]` profiles over manually setting only the viewport.
- Keep one reusable test whenever possible; add mobile-specific steps only when needed.
- Avoid maintaining separate desktop and mobile test suites unless the experiences are substantially different.
- Use `testInfo.project.name` to apply project-specific logic.
- Run responsive tests in your CI/CD pipeline across representative desktop and mobile projects.

---

# Key Takeaways

- Playwright provides built-in device emulation through the `devices` library.    
- Create separate **projects** for mobile testing rather than duplicating configuration.
- Device emulation includes viewport, user agent, touch support, and other mobile characteristics.
- Use `testInfo.project.name` to make a single test work across desktop and mobile when only minor differences exist.
- For enterprise projects, run the same tests across multiple browser and device projects to maximize coverage while minimizing duplicated test code.
