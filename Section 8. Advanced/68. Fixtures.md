# Overview

**Fixtures** are one of the most powerful features in Playwright.

A fixture is responsible for **setting up everything a test needs before execution** and **cleaning up afterward**.

Instead of writing repeated setup code inside every test or relying only on `beforeEach()` and `afterEach()`, Playwright fixtures allow you to create reusable, isolated, and dependency-aware setup logic.

The official Playwright philosophy is:

> **"Give each test everything it needs and nothing else."**

This ensures that:

- Tests remain independent.
- Tests are easier to read.
- Setup logic is reusable.
- Cleanup happens automatically.
- Test execution becomes faster and more maintainable.

---

# Why Do We Need Fixtures?

Consider this test:

```ts
test.beforeEach(async ({ page }) => {
    await page.goto("/");
});

test("Submit Form", async ({ page }) => {
    const pm = new PageManager(page);

    await pm.navigateTo().formLayoutsPage();

    await pm.onFormLayoutsPage().submitGridForm(...);

    await pm.onFormLayoutsPage().submitInlineForm(...);
});
```

Every test repeats:

- Opening the application    
- Creating PageManager
- Navigating to Forms Layout page

These are **preconditions**, not the actual test.

The real validation is only:

```ts
await pm.onFormLayoutsPage().submitGridForm(...);

await pm.onFormLayoutsPage().submitInlineForm(...);
```

Everything else is environment setup.

Fixtures allow us to move this setup outside the test.

# beforeEach vs Fixtures

### beforeEach

```ts
test.beforeEach(async ({ page }) => {
    await page.goto("/");
});
```

Good for:

- Simple setup    

Limitations:

- Difficult to reuse
- Cannot easily share objects
- Cannot define dependencies
- Less flexible

---

### Fixtures

```ts
test("My Test", async ({ pageManager }) => {

})
```

The fixture already provides:

- initialized objects    
- prepared environment
- automatic cleanup

# What Is a Fixture?

A fixture is a function that:

1. Creates resources.    
2. Makes them available to the test.
3. Cleans them afterward.

General structure:

```ts
fixture

  ↓

Setup

  ↓

await use(resource)

  ↓

Teardown
```

Everything before `use()` is **setup**.

Everything after `use()` is **cleanup**.

---

# Creating a Fixture

Playwright fixtures are created by extending the default `test` object.

```ts
import { test as base } from "@playwright/test";

export const test = base.extend({

});
```

`base` is the original Playwright test object.

`extend()` adds custom fixtures.

---

# Example: Form Layout Fixture

Suppose every test needs to navigate to the Forms Layout page.

Instead of writing:

```ts
await page.goto("/");

await page.getByText("Forms").click();

await page.getByText("Form Layouts").click();
```

Create a fixture.

```ts
formLayoutsPage: async ({ page }, use) => {

    await page.goto("/");

    await page.getByText("Forms").click();

    await page.getByText("Form Layouts").click();

    await use("");

}
```

# Understanding use()

This is the most important part.

```ts
await use(value);
```

Think of it as:

```text
Setup

  ↓

Pause

  ↓

Run Test

  ↓

Resume

  ↓

Cleanup
```

Everything before:

```ts
await use()
```

runs before the test.

Everything after:

```ts
await use()
```

runs after the test.

# Using the Fixture

Instead of importing Playwright's test:

```ts
import { test } from "@playwright/test";
```

Import your extended version.

```ts
import { test } from "../test-options";
```

Then simply request the fixture.

```ts
test("My Test", async ({ formLayoutsPage }) => {

});
```

Playwright automatically initializes it.

---

# Automatic Fixtures

Normally:

```ts
test("Example", async ({ formLayoutsPage }) => {

});
```

The fixture only runs if requested.

You can make it automatic.

```ts
{
    auto: true
}
```

Example:

```ts
formLayoutsPage: [

async ({ page }, use) => {

...

},

{

auto: true

}

]
```

Now it runs before every test automatically.

No need to inject:

```ts
({ formLayoutsPage })
```

---

# Creating a PageManager Fixture

Instead of:

```ts
const pm = new PageManager(page);
```

inside every test,

create a fixture.

```ts
pageManager: async ({ page }, use) => {

    const pm = new PageManager(page);

    await use(pm);

}
```

Now every test simply receives it.

```ts
test("Example", async ({ pageManager }) => {

});
```

Much cleaner.

# Fixture Dependencies

Fixtures can depend on other fixtures.

Example:

```ts
pageManager:
async (

{

page,

formLayoutsPage

},

use

) => {

...
}
```

Execution order:

```text
formLayoutsPage

  ↓

pageManager

  ↓

Test
```

Playwright resolves dependencies automatically.

# Execution Flow

Suppose we have:

```text
Fixture A

  ↓

Fixture B

  ↓

Test
```

Execution becomes:

```text
Fixture A Setup

  ↓

Fixture B Setup

  ↓

Test

  ↓

Fixture B Cleanup

  ↓

Fixture A Cleanup
```

Notice:

Cleanup happens in reverse order.

Exactly like a stack.

# Setup vs Teardown

Example:

```ts

    await use(page);

    await logout();
```

Execution:

```text

  ↓

Run Test

  ↓

Logout
```

Very powerful.

# Why Fixtures Are Faster

The instructor observed that the fixture version executed faster than the `beforeEach` version.

This is because Playwright can optimize fixture initialization and dependency resolution. Fixtures are also lazily initialized (only when needed) and integrated into Playwright's execution engine.

> **Important:** The performance gain depends on the scenario. Fixtures are primarily a design and maintainability feature—not a guaranteed performance optimization. The main benefit is cleaner, reusable, and isolated test setup.

# Fixtures vs Hooks

| Feature                   | beforeEach | Fixture   |
| ------------------------- | ---------- | --------- |
| Setup                     | ✅          | ✅         |
| Cleanup                   | Limited    | ✅         |
| Dependency Injection      | ❌          | ✅         |
| Automatic Object Creation | ❌          | ✅         |
| Reusable                  | Limited    | ✅         |
| Can Return Objects        | ❌          | ✅         |
| Dependency Management     | ❌          | ✅         |
| Isolation                 | Moderate   | Excellent |

# Best Practices

- Use fixtures for reusable setup that multiple tests need.    
- Keep fixtures focused on a single responsibility (e.g., authentication, page manager, test data).
- Avoid putting business assertions inside fixtures—fixtures should prepare the environment, not verify behavior.
- Use fixture dependencies instead of duplicating setup logic.
- Use `auto: true` sparingly; automatic fixtures run for every test and can slow down suites if overused.
- Prefer meaningful fixture names such as `authenticatedPage`, `pageManager`, or `adminUser`.
- Keep teardown logic after `await use(...)` to ensure resources are cleaned up even if the test fails.

# Key Takeaways

- Fixtures are Playwright's preferred mechanism for setting up and tearing down test environments.    
- They are created by extending the base `test` object with `test.extend()`.
- Everything before `await use()` is setup; everything after it is teardown.
- Fixtures can provide reusable objects (such as `PageManager` or API clients) directly to tests.
- Fixtures can depend on other fixtures, and Playwright resolves the dependency order automatically.
- Cleanup executes in the reverse order of setup.
- Compared to `beforeEach`/`afterEach`, fixtures provide better isolation, dependency injection, reusability, and maintainability.
- In enterprise automation frameworks, fixtures are commonly used for authentication, page objects, API clients, database connections, and test data management.
