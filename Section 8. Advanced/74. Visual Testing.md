# What is Visual Testing?

Visual testing (also called **Visual Regression Testing**) verifies that the application's UI looks exactly as expected.

Instead of checking:

- text    
- attributes
- visibility
- CSS values

Playwright simply compares **screenshots**.

If the current screenshot differs from the stored baseline image, the test fails.

Example:

Expected UI:

```text
------------------------
 Name
 Email

 ( ) Option 1
 ( ) Option 2
------------------------
```

Current UI:

```text
------------------------
 Name
 Email

 ( ) Option 2
 ( ) Option 1
------------------------
```

Even though the functionality works correctly, the UI changed.

Visual testing catches this immediately.

---

# Why Use Visual Testing?

Visual assertions are excellent for detecting:

- Broken layouts    
- Missing buttons
- Incorrect colors
- Wrong fonts
- UI alignment issues
- CSS regressions
- Unexpected design changes

Instead of writing many assertions like:

```ts
await expect(button).toBeVisible();
await expect(button).toContainText("Submit");
await expect(button).toHaveCSS("color", "blue");
await expect(button).toHaveCSS("font-size", "16px");
```

One visual assertion can validate everything:

```ts
await expect(button).toHaveScreenshot();
```

# Creating Your First Visual Assertion

Playwright provides:

```ts
toHaveScreenshot()
```

Example:

```ts
await expect(usingTheGridForm).toHaveScreenshot();
```

Notice:

We pass a **Locator**, not the Page.

Playwright will capture only that specific element.

This image becomes the **expected UI**.

Example:


Playwright saves this image.

# How Comparison Works

```text

        │
        ▼

        │
        ▼

        │
        ▼

```

```text

  +

Test Name

  +

  +

.png
```

Example:

```text

		↓

```

If you execute the same test in Firefox:

```text

```

Different browsers render pages slightly differently.

Therefore each project maintains its own snapshot.

# Pixel Differences

Sometimes tiny rendering differences occur.

Examples:

- font rendering    
- anti-aliasing
- animations
- shadows
- browser rendering

These differences shouldn't fail your test.

Playwright allows tolerance.

---

# maxDiffPixels

You can configure:

```ts
await expect(locator).toHaveScreenshot({
    maxDiffPixels: 150,
});
```

Meaning:

Ignore differences up to 150 pixels.

Example:

Difference:

```text
140 pixels
```

Result:

```text
PASS
```

Difference:

```text
220 pixels
```

Result:

```text
FAIL
```

---

# Global Configuration

Instead of configuring every assertion:

```ts
await expect(locator).toHaveScreenshot({
    maxDiffPixels: 150
});
```

You can configure it once.

Example:

```ts
export default defineConfig({

    expect: {

        toHaveScreenshot: {
            maxDiffPixels: 50,
        },

    },

});
```

Now every screenshot assertion uses this value.

# Updating Snapshots

Suppose your UI has intentionally changed.

Old UI

```text
Blue Button
```

New UI

```text
Green Button
```

Your tests fail because the baseline is outdated.

Instead of deleting screenshots manually, run:

```bash
npx playwright test --update-snapshots
```

Playwright will:

- execute all visual tests
- capture new screenshots
- replace old baseline images

This updates the expected UI for future runs.

# Real-World Uses

Visual testing is commonly used to detect:

- Broken page layouts    
- CSS regressions
- Missing UI components
- Incorrect spacing
- Font changes
- Responsive design issues
- Theme (Light/Dark mode) regressions
- Localization UI issues
- Browser rendering inconsistencies
- Design system validation

Many teams integrate visual testing into CI/CD pipelines so UI regressions are caught before deployment.

---

# Best Practices

- Use visual testing only for UI validation, not business logic.    
- Capture the smallest meaningful element instead of the entire page to reduce flakiness.
- Keep animations disabled or stabilized during visual tests.
- Set reasonable `maxDiffPixels` or tolerance values to avoid failures from minor rendering differences.
- Maintain separate snapshots for different browsers when cross-browser testing.
- Review snapshot updates carefully before committing them to version control.
- Use `--update-snapshots` only after confirming that UI changes are intentional.
- Store baseline screenshots in version control so all team members use the same expected images.

---

# Key Takeaways

- Visual testing compares screenshots instead of DOM properties.    
- `toHaveScreenshot()` automatically creates a baseline image on the first run.
- Future executions compare the current screenshot against the stored baseline.
- On failure, Playwright generates **expected**, **actual**, and **diff** images for debugging.
- The HTML report provides an interactive comparison slider.
- `maxDiffPixels` helps reduce failures caused by small rendering differences.
- Global screenshot comparison settings can be configured in `playwright.config.ts`.
- `npx playwright test --update-snapshots` regenerates all baseline screenshots after intentional UI changes.
