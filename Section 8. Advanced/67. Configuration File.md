# Overview

The **Playwright configuration file (`playwright.config.ts`)** is the central place where all framework settings are managed.

Instead of repeating configuration across multiple tests, Playwright allows you to configure everything once:

- Browser settings    
- Timeouts
- Retries
- Reporters
- Screenshots
- Videos
- Traces
- Parallel execution
- Authentication
- Base URLs
- Projects
- Environment-specific settings
- Custom configuration files

Think of it as the **brain of the Playwright framework**.

---

# Recommended Practice

A newly generated Playwright config contains many comments.

Example:

```ts
// Base URL
// Maximum timeout
// Retry configuration
// Worker configuration
// etc.
```

For a production project:

- Remove unnecessary comments.
- Keep only settings relevant to your framework.
- Make the config clean and readable.

# Example Clean Configuration

```ts
import { defineConfig } from "@playwright/test";

export default defineConfig({

    timeout: 40000,

    retries: 1,

    reporter: "html",

    use: {

        baseURL:
            process.env.URL || "http://localhost:4200",

        trace: "on-first-retry",

        video: "off",

        navigationTimeout: 5000,
    },

    projects: [

        {
            name: "chromium",
        },

        {
            name: "firefox",
            use: {
                browserName: "firefox",
            },
        },
    ],
});
```

# Two Main Sections

The configuration is divided into two major parts.

## 1. Global Configuration

Applies to **every project** unless overridden.

Example:

```ts
timeout: 40000,

retries: 1,

reporter: "html"
```

These settings affect all browsers and all tests.

---

## 2. Projects

Each project can override global settings.

Example:

```ts
projects: [

{
    name: "chromium"
},

{
    name: "firefox"
}

]
```

Projects are **not limited to browsers**.

They can represent:

- browsers
- environments
- smoke tests
- regression suite
- mobile tests
- API tests
- UI tests

Projects are simply **logical groups of configuration**.

# Global Settings

Common global properties include:

## timeout

Maximum execution time for a test.

```ts
timeout: 40000
```

40 seconds.

---

## retries

Retry failed tests.

```ts
retries: 1
```

---

## reporter

Specifies how test results are reported.

Example:

```ts
reporter: "html"
```

Other reporters:

```ts
"list"

"dot"

"line"

"json"

"junit"

"html"
```

---

## workers

Maximum parallel workers.

```ts
workers: 5
```

or

```ts
workers: 1
```

to disable parallel execution.

---

## fullyParallel

```ts
fullyParallel: true
```

Runs tests inside a spec file in parallel.

# use Block

The `use` section contains runtime settings shared by tests.

Example:

```ts
use: {

    baseURL: "...",

    viewport: {...},

    trace: "...",

    screenshot: "...",

    video: "...",

}
```

---

## baseURL

Instead of

```ts
await page.goto("http://localhost:4200");
```

Use

```ts
await page.goto("/");
```

Configuration

```ts
use: {

baseURL: "http://localhost:4200"

}
```

---

environment variable.

# Project-Level Configuration

A project can override any global setting.

Example

Global:

```ts
video: "off"
```

Firefox project:

```ts
projects: [

	{
	
		name: "firefox",
		
		use: {
		
			video: "on"
		
		}
	
	}

]
```

Result

```text
Chromium

	↓

No video

Firefox

	↓

Video enabled
```

Project configuration always overrides global configuration.

---


Firefox

```ts

```

Firefox now has

```text

```

Chromium still has

# testMatch

Limits a project to specific tests.

```ts
testMatch:

"usePageObjects.spec.ts"
```

Can also use patterns:

```ts
testMatch:

/.*api.*/
```

or

```ts
testMatch:

"**/*smoke*.spec.ts"
```

```ts
import { defineConfig } from "@playwright/test";

	use:{
	
		baseURL:
		
		"https://myproductionapp.com"
	
	},

		{
		
			name:"chromium"
		
		}
		
	]

});
```

Very small configuration.

Everything else uses Playwright defaults.

# Running a Custom Config

Instead of:

```bash
npx playwright test
```

Use

```bash
npx playwright test --config=playwright.prod.config.ts
```

Now Playwright uses the production configuration.

# Best Practices

- Keep `playwright.config.ts` clean and readable.
- Store shared settings globally.
- Override settings only when necessary at the project level.
- Use environment variables instead of hardcoded URLs or credentials.
- Use multiple configuration files for different environments (dev, QA, staging, production) when they differ significantly.
- Use projects to organize test suites logically—not just by browser.
- Keep secrets (tokens, passwords, API keys) out of the configuration file; load them securely through environment variables or a secrets manager.
- Prefer descriptive project names (`smoke-chromium`, `api-tests`, `mobile-web`) over generic names when projects serve different purposes.

---

# Key Takeaways

- `playwright.config.ts` is the central configuration file for the Playwright framework.
- Configuration is organized into **global settings**, the **`use` block**, and **projects**.
- Project-specific settings override global settings.
- Projects can represent browsers, environments, or any logical grouping of tests.
- The `use` block controls runtime behavior such as `baseURL`, `trace`, `video`, `viewport`, and `storageState`.
- `testMatch` and `testIgnore` allow projects to target specific test files.
- Playwright supports multiple configuration files, which can be selected using `--config`.
- A clean, modular configuration makes large automation frameworks easier to maintain and scale.
