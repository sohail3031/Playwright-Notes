# Overview

In large automation projects, not every test failure is caused by an application bug. Tests may fail due to temporary issues such as:

- Network latency    
- Slow API responses
- Timing issues
- Browser rendering delays
- Third-party service outages
- Temporary infrastructure problems

These failures are called **flaky tests**.

To improve test stability, Playwright provides a built-in **Retry Mechanism** that automatically reruns failed tests.

---

# What is a Retry?

A **retry** is an automatic re-execution of a failed test.

Example:

```bash
Test Run #1
------------
Login Test       вњ… Pass
Create User      вќЊ Fail
Delete User      вњ… Pass
```

If retries are enabled:

```bash
Retry #1
---------
Create User      вњ… Pass
```

Final Result:

```bash
Login Test       Passed
Create User      Passed (after retry)
Delete User      Passed
```

---

# Why Do Retries Exist?

Not every failure is an actual bug.

Example:

```bash
Expected: Welcome John

Actual:
Loading...
```

The application was simply slower than usual.

Running the test again often succeeds.

Retries help distinguish between:

- Temporary failures
- Real application defects

---

# Understanding Playwright Workers

Before understanding retries, it's important to understand **Workers**.

A **Worker** is a separate Playwright process that executes one or more tests.

Each worker launches:

- A fresh browser instance
- A clean browser context
- An isolated execution environment

This prevents state leakage between tests.

# Why Does Playwright Create a New Worker?

This is one of Playwright's biggest advantages.

Suppose a test:

- logs out
- deletes cookies
- corrupts local storage
- crashes the browser
- changes application state

Retrying inside the same browser could fail again for unrelated reasons.

Instead, Playwright creates:

- new browser
- new context
- fresh storage
- fresh cookies
- fresh session

This guarantees test isolation.

# Configuring Retries

Inside `playwright.config.ts`

```ts
use: {
    ...
},

retries: 1,
```

Meaning:

```bash
Run failed test one additional time.
```

---

# CI vs Local Configuration

Playwright's default template contains:

```ts
retries: process.env.CI ? 2 : 0,
```

Meaning:

If running inside CI:

```bash
Retries = 2
```

Otherwise:

```bash
Retries = 0
```

### Why?

Locally:

Developers usually want failures immediately.

CI:

Infrastructure is more unstable.

Retries reduce false failures.

# Overriding Retries for Specific Tests

Global:

```ts
retries: 1
```

But one test suite is especially flaky.

Override:

```ts
test.describe.configure({
    retries: 2
});
```

Now every test inside this `describe` block gets:

```bash
Retry #1

Retry #2
```

instead of

```bash
Retry #1
```

---

# Example

```ts
test.describe("Checkout", () => {

});
```

Configure:

```ts
test.describe.configure({
    retries: 2
});
```

Applies only to this group.

Other tests continue using:

```bash
retries: 1
```

# Real-World Use Case

Suppose your test creates data.

Retrying immediately causes duplicates.

Before retry:

```ts
if (testInfo.retry) {
    await deleteTestUser();
}
```

or

```ts
if (testInfo.retry) {
    await cleanDatabase();
}
```

or

```ts
if (testInfo.retry) {
    await resetEnvironment();
}
```

This makes retries much more reliable.

# Understanding Test Status

Playwright reports different outcomes based on retry behavior:

|Initial Result|Retry Result|Final Status|
|---|---|---|
|Pass|вЂ”|Passed|
|Fail|Pass|Flaky (Passed after Retry)|
|Fail|Fail|Failed|

A **flaky** test is one that passes only after one or more retries. This indicates the test or application behavior may be unstable and should be investigated.

# Best Practices

## Keep retries low

Good:

```ts
retries: 1
```

or

```ts
retries: 2
```

Avoid:

```ts
retries: 10
```

This only hides underlying issues and increases execution time.

---

## Fix flaky tests

Retries are a temporary safeguardвЂ”not a replacement for improving test quality.

Common causes of flaky tests include:

- Poor synchronization
- Unstable locators
- Shared test data
- Network dependencies
- Timing issues

Address these root causes instead of relying on retries.

---

## Use retries mainly in CI

Recommended configuration:

```ts
retries: process.env.CI ? 2 : 0
```

This provides faster local feedback while making CI pipelines more resilient to transient failures.

---

## Clean up before retry

If your test modifies external state (database, files, APIs), use `testInfo.retry` to reset the environment before the retry.

---

## Don't use retries to hide bugs

If a test consistently passes only after retries, investigate the cause. Excessive reliance on retries can mask genuine defects and reduce confidence in the test suite.

---

# Common Mistakes

### Setting very high retry counts

```ts
retries: 10
```

This slows down execution and conceals unstable tests.

---

### Assuming retries fix all failures

Retries only help with **temporary** issues.

They won't fix:

- Broken locators
- Incorrect assertions
- Real application defects
- Invalid test logic

---

### Ignoring flaky tests

A test that repeatedly requires retries should be analyzed and stabilized rather than accepted as "good enough."

---

# Advantages of Playwright Retries

- Automatic retry of failed tests
- Fresh worker and browser context for each retry
- Improved CI pipeline stability
- Better isolation between tests
- Easy global and per-suite configuration
- Built-in support via `testInfo.retry`
- No additional libraries required

---

# Key Takeaways

- A **retry** automatically reruns a failed test.
- Playwright starts a **new worker** for every retry, ensuring a clean browser session.
- Configure retries globally using the `retries` option in `playwright.config.ts`.
- Override retry behavior for specific suites with `test.describe.configure({ retries: ... })`.
- Use `testInfo.retry` to detect retries and perform cleanup or setup before rerunning a test.
- Retries improve reliability but should complementвЂ”not replaceвЂ”well-designed, stable automation tests.
