# Difference Between Hooks, Fixtures, and Project Setup

| Feature          | Runs                          | Best For                     |
| ---------------- | ----------------------------- | ---------------------------- |
| beforeEach       | Before every test             | Test-specific setup          |
| beforeAll        | Once per test file            | Shared setup within one spec |
| Fixture          | Before a test that needs it   | Reusable test dependencies   |
| Project Setup    | Once before an entire project | Environment initialization   |
| Project Teardown | Once after the entire project | Environment cleanup          |

---

# Advantages

- Eliminates duplicate setup code.    
- Keeps the test database clean.
- Speeds up large test suites.
- Makes tests deterministic and repeatable.
- Allows chaining multiple setup stages (authentication → seed data → tests → cleanup).
- Integrates naturally with Playwright's project dependency system.

---

# Best Practices (Industry)

1. Keep setup scripts focused on **one responsibility** (authentication, seed data, mock server, etc.). 
2. Always pair setup with a teardown when creating persistent resources.
3. Share identifiers (such as article slugs or IDs) through controlled mechanisms like environment variables or temporary files only when appropriate.
4. Make setup scripts **idempotent** where possible, so rerunning them doesn't leave the system in a bad state.
5. Use project setup for **environment-level preparation**, not for per-test initialization—that's what fixtures and hooks are for.
6. Keep setup fast; avoid creating unnecessary data.

# Interview Questions

### Q1. Why use Project Setup instead of `beforeEach()`?

**Answer:** Project setup runs once for a project and is ideal for preparing shared test environments, while `beforeEach()` runs before every test and is intended for test-specific initialization.

---

### Q2. What is a project dependency?

**Answer:** A dependency ensures one project completes before another starts. This is useful for workflows such as authentication → data seeding → test execution.

---

### Q3. What is project teardown?

**Answer:** Project teardown executes after the dependent project finishes, allowing cleanup such as deleting test data, stopping services, or resetting the environment.

---

### Q4. Can projects depend on other projects?

**Answer:** Yes. Playwright supports chaining project dependencies, enabling complex setup pipelines like authentication → seed data → execute tests.

---

### Q5. What kinds of tasks belong in project setup?

**Answer:** Tasks that prepare the overall test environment, such as generating authentication tokens, seeding databases, creating shared test entities, or starting required infrastructure.
