Environment variables allow you to **store configuration values outside your test code** so the same automation framework can run against multiple environments (Development, QA, Staging, Production) without modifying the test files.

Instead of hardcoding URLs, credentials, API endpoints, or secrets directly in your tests, they should be stored as environment variables.

---

# Why Environment Variables?

Suppose you write:

```ts
await page.goto("http://localhost:4200");
```

This works only for your local machine.

Tomorrow you may need to run the same test against:

- Development
- QA
- Staging
- Production

Without environment variables, you would have to manually edit every test.

Instead:

```text
  Test
    ↓
Reads Base URL
    ↓
Environment Variable
    ↓
Current Environment
```

This keeps your framework clean and scalable.

---

# Types of Environment Variables in Playwright

The instructor demonstrates **five different approaches**:

1. Built-in `baseURL`    
2. Project-specific environment variables
3. Custom Playwright fixtures (Test Options)
4. `.env` file with `dotenv`
5. Command-line environment variables

---

# 1. Playwright Built-in baseURL

Playwright already supports one environment variable called **baseURL**.

Instead of:

```ts
await page.goto("http://localhost:4200");
```

put the URL inside `playwright.config.ts`.

```ts
use: {
    baseURL: "http://localhost:4200"
}
```

Then your tests become

```ts
await page.goto("/");
```

Playwright automatically combines

```text
baseURL
+
/
```

Result

```http
http://localhost:4200/
```

---

## Benefits

Instead of changing every test

```text
50 Tests

  ↓
  
50 URLs
```

You only change

```text
playwright.config.ts

  ↓

baseURL
```

One change updates the entire project.

---

# Example

Before

```ts
await page.goto("http://localhost:4200");
```

After

```ts
// playwright.config.ts

use:{
    baseURL:"http://localhost:4200"
}
```

Test

```ts
await page.goto("/");
```

Much cleaner.

# 2. Different Base URLs per Project

Playwright projects are not limited to browsers.

Most beginners think:

```text
Projects

Chromium
Firefox
Webkit
```

Actually, projects can represent:

- Browsers
- Environments
- Regions
- Devices

Example

```ts
projects:[
{
    name:"dev",

    use:{
        baseURL:"http://localhost:4201"
    }
},

{
    name:"staging",

    use:{
        baseURL:"http://localhost:4202"
    }
}
]
```

Now you can run

```bash
npx playwright test --project=dev
```

or

```bash
npx playwright test --project=staging
```

Each project automatically uses its own URL.

---

# Project Override

Playwright follows this priority:

```text
Project baseURL

	↓

Global baseURL
```

If both exist

```text
Global

  ↓

Ignored

  ↓

Project value used
```

Project settings override global settings.

# 3. Creating Your Own Environment Variables (Test Options)

Suppose your framework needs more variables than just baseURL.

For example

```text
dragDropURL

apiURL

loginURL

adminURL
```

Playwright does **not** automatically recognize arbitrary keys.

This will **NOT** work:

```ts
use:{
    globalURL:"..."
}
```

You must extend Playwright.

---

## Step 1

Create

```text
test-options.ts
```

---

## Step 2

Import Playwright test

```ts
import { test as base } from "@playwright/test";
```

---

## Step 3

Create custom type

```ts
export type TestOptions = {

    globalURL:string;

};
```

---

## Step 4

Extend Playwright

```ts
export const test = base.extend<TestOptions>({

    globalURL:["",{option:true}]

});
```

Meaning

```text
Create a new fixture

called

globalURL
```

---

## Step 5

Tell Playwright about this new type

Inside

```ts
playwright.config.ts
```

```ts
import type { TestOptions } from "./test-options";
```

Then

```ts
export default defineConfig<TestOptions>({
```

Now Playwright understands

```text
globalURL
```

---

## Step 6

Assign value

```ts
use:{
    globalURL:"https://example.com"
}
```

---

## Step 7

Use inside tests

Instead of

```ts
import { test } from "@playwright/test";
```

Import

```ts
import { test } from "../test-options";
```

Then

```ts
test("Example", async ({ page, globalURL }) => {

    await page.goto(globalURL);

});
```

---

## Flow

```text
playwright.config.ts

	↓

globalURL

	↓

test-options.ts

	↓

Playwright Fixture

	↓

  Test
```

# 4. Using .env File

This is the **industry standard**.

Most companies store configuration inside

```text
.env
```

Example

```text
.env

USERNAME=test@test.com

PASSWORD=Welcome1

API_URL=https://api.dev.com

BASE_URL=https://dev.company.com
```

---

# Why?

Never hardcode secrets.

Bad

```ts
password:"Welcome1"
```

Good

```ts
password:process.env.PASSWORD
```

---

# Installing dotenv

Playwright doesn't automatically read `.env`.

Install

```bash
npm install dotenv --save-dev
```

---

# Enable dotenv

At the top of

```text
playwright.config.ts
```

```ts
require("dotenv").config();
```

Now Playwright loads every variable from `.env`.

---

# Access Variables

```ts
process.env.USERNAME
```

```ts
process.env.PASSWORD
```

```ts
process.env.BASE_URL
```

Example

```ts
await page.fill(

"#email",

process.env.USERNAME!

);
```

---

# Environment Variables are Strings

Everything inside `.env` is stored as text.

Example

```text
TIMEOUT=5000
```

Actually becomes

```text
"5000"
```

Convert if needed

```ts
Number(process.env.TIMEOUT)
```

# 5. Passing Variables from CLI

You don't even need `.env`.

Example

```bash
URL=https://google.com npx playwright test
```

Then

```ts
await page.goto(process.env.URL!);
```

Playwright reads it directly.

Very useful for CI/CD.

---

# Saving CLI Variables in NPM Scripts

Instead of typing

```bash
URL=https://staging.com npx playwright test
```

Create

```json
"scripts":{

"staging":"URL=https://staging.com npx playwright test"

}
```

Run

```bash
npm run staging
```

Much easier.

Example

```ts

```

Now changing `.env`

```text
BASE_URL=https://dev.com
```

or

```text
BASE_URL=https://prod.com
```

changes the entire framework.

# Ternary Operator for Environment Selection

Instead of projects

```ts
baseURL:
process.env.DEV=="1"

?

"http://localhost:4201"

:

process.env.STAGING=="1"

?

"http://localhost:4202"

:

"http://localhost:4200";
```

Run

```bash
DEV=1 npx playwright test
```

or

```bash
STAGING=1 npx playwright test
```

Playwright automatically chooses the correct URL.

# Credentials from .env

Instead of

```ts
.fill("test@test.com")
```

Use

```ts
.fill(process.env.USERNAME!)
```

Password

```ts
.fill(process.env.PASSWORD!)
```

This keeps credentials out of source code.

# Never Commit .env

`.env` usually contains

- Passwords
- Tokens
- Database credentials
- API Keys
- Secrets

Always add

```text
.gitignore
```

```text
.env
```

This prevents secrets from being pushed to GitHub.

# Real-World Best Practices

A typical enterprise Playwright framework uses the following structure:

```text
project
│
├── playwright.config.ts
├── .env
├── .env.dev
├── .env.qa
├── .env.staging
├── .env.prod
│
├── test-options.ts
│
├── tests/
│
├── pages/
│
└── utils/
```

Each environment file contains its own configuration:

```text
.env.dev
BASE_URL=https://dev.company.com
API_URL=https://dev-api.company.com
USERNAME=dev_user
PASSWORD=dev_password
```

```text
.env.qa
BASE_URL=https://qa.company.com
API_URL=https://qa-api.company.com
USERNAME=qa_user
PASSWORD=qa_password
```

```text
.env.staging
BASE_URL=https://staging.company.com
API_URL=https://staging-api.company.com
USERNAME=staging_user
PASSWORD=staging_password
```

```text
.env.prod
BASE_URL=https://company.com
API_URL=https://api.company.com
USERNAME=prod_user
PASSWORD=prod_password
```

A common pattern is to load the appropriate file based on an environment variable:

```ts
import dotenv from "dotenv";

dotenv.config({
  path: `.env.${process.env.TEST_ENV || "dev"}`,
});
```

Then run:

```bash
TEST_ENV=qa npx playwright test
```

or

```bash
TEST_ENV=staging npx playwright test
```

This approach avoids modifying `playwright.config.ts` when switching environments.

# Interview Questions

### Why use environment variables?

To separate configuration from code, allowing the same tests to run against different environments without changes.

### What is `baseURL`?

A built-in Playwright configuration that lets you call `page.goto("/")` instead of hardcoding full URLs.

### Why use `.env` files?

To store environment-specific values such as URLs, usernames, passwords, API keys, and other secrets outside the source code.

### Why should `.env` be added to `.gitignore`?

Because it often contains sensitive information that should never be committed to version control.

### How do you access environment variables?

```ts
process.env.VARIABLE_NAME
```

### Can Playwright have custom environment variables?

Yes. You can create custom fixtures using `base.extend()` (as shown with `test-options.ts`) or use `process.env` variables loaded from `.env` files or the command line.

### Which approach is most common in industry?

The most common approach is using **`dotenv` with multiple `.env` files** (for example, `.env.dev`, `.env.qa`, `.env.staging`, `.env.prod`) combined with `process.env` in `playwright.config.ts`. This is widely adopted because it cleanly separates configuration from code, keeps secrets out of the repository, and integrates well with CI/CD pipelines.
