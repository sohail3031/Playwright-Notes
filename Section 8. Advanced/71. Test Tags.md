# What are Test Tags?

Test tags are **labels** attached to tests or groups of tests.

They allow you to execute only a subset of tests without changing the test code.

Instead of running every test, you can run:

- Smoke tests    
- Regression tests
- Sanity tests
- API tests
- UI tests
- Authentication tests
- Payment tests
- Critical tests
- Mobile tests
- etc.

---

# Why use Tags?

Imagine a project with

```text
4,500 Playwright tests
```

Every commit does **not** need to execute all 4,500 tests.

Instead,

```text
Developer Commit
        ↓
Run Smoke Tests (30)

Nightly Build
        ↓
Run Regression Tests (4500)

Hotfix Deployment
        ↓
Run Payment Tests

Release Candidate
        ↓
Run Smoke + Regression

Authentication Changes
        ↓
Run Authentication Tests
```

This is exactly why tags exist.

# Without Tags

```ts
tests/

login.spec.ts
checkout.spec.ts
cart.spec.ts
payment.spec.ts
orders.spec.ts
profile.spec.ts
```

How do you execute only payment tests?

Without tags you would have to

- rename files    
- move files
- create separate folders

Not scalable.

---

# With Tags

```ts
login.spec.ts

@smoke
@auth

checkout.spec.ts

@regression
@payment

payment.spec.ts

@payment
@critical
@smoke
```

Now Playwright can execute exactly what you need.

```bash
npx playwright test --grep "@smoke"
```

---

```bash

```

---

```bash

```

# Running Multiple Tags

Suppose

```ts
@smoke
@payment
```

Run both

Mac/Linux

```bash
npx playwright test --grep "@smoke|@payment"
```

Windows CMD

```bash
npx playwright test --grep "@smoke^|@payment"
```

PowerShell

```powershell
npx playwright test --grep "@smoke --% | @payment"
```

# Tags inside describe()

Instead of tagging every test individually

```ts
test.describe("Authentication @auth", () => {

    test("Login", async () => { });

    test("Logout", async () => { });

    test("Forgot Password", async () => { });

});
```

Every test inside inherits the tag.

---

# Example

```ts
test.describe("Checkout @payment", () => {

    test("Credit Card", async () => { });

    test("PayPal", async () => { });

    test("Apple Pay", async () => { });

});
```

Running

```bash
--grep @payment
```

executes all three tests.

# Multiple Tags

```ts
test(

});
```

This belongs to

- Smoke suite    
- Authentication suite

# Excluding Tags

Use

```bash
--grep-invert
```

Example

```bash
npx playwright test --grep-invert "@slow"
```

Runs every test except slow tests.

---

Exclude flaky tests

```bash
npx playwright test --grep-invert "@flaky"
```

---

Exclude mobile

```bash
npx playwright test --grep-invert "@mobile"
```

# Best Practices

✅ Use consistent naming conventions:

```ts
@smoke
@regression
@critical
@auth
@payment
@api
@ui
@mobile
@desktop
@visual
@security
@performance
```

Avoid inconsistent variants like:

```ts
@Smoke
@SMOKE
@SmokeTests
```

---

## Keep Tags Meaningful

Good

```ts
@payment
@checkout
@auth
```

Bad

```ts
@test1
@random
@featureA
```

---

## One Test Can Have Multiple Tags

```ts
@smoke
@payment
@critical
```

This is common and encouraged.

---

## Don't Over-Tag

Avoid attaching every possible tag to every test. Use only tags that genuinely help organize and select the test.

---

```text
  ↓
```

This creates highly flexible execution strategies.

# Advantages

- Execute only relevant tests.    
- Reduce CI/CD execution time.
- Organize large test suites effectively.
- Improve deployment confidence.
- Enable feature-specific testing.
- Support multiple pipelines (smoke, regression, critical, etc.).
- Easy integration with Jenkins, GitHub Actions, Azure DevOps, GitLab CI, and similar platforms.

---

# Key Takeaways

- **Tags** are labels that categorize tests.    
- Use `--grep` to run only tests with matching tags.
- Use `--grep-invert` to exclude tagged tests.
- Tags can be applied to individual tests or entire `describe()` blocks.
- Multiple tags per test are common in enterprise projects.
- Tags are a cornerstone of scalable test execution strategies, especially in CI/CD pipelines where different workflows (smoke, regression, critical, feature-specific) need to run independently.
