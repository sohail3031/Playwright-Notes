# Overview

As test suites grow larger, **execution time becomes a major concern**. Running hundreds or thousands of tests sequentially can take hours. To solve this problem, Playwright provides **parallel execution**, allowing multiple tests to run simultaneously using independent browser instances called **workers**.

Parallel execution is one of Playwright's biggest advantages because it significantly reduces overall test execution time while maintaining test isolation.

> **Key Idea:** Every worker is an independent browser process running in a clean incognito context.

---

## What is a Worker?

A **worker** is an independent Playwright process that:

- Launches its own browser instance    
- Uses a fresh browser context (Incognito)
- Executes assigned tests independently
- Does not share browser state with other workers

Because each worker is isolated, tests running in parallel do not interfere with each other.

---

## Default Parallel Execution

Suppose your project contains five spec files:

```bash
tests/
│
├── login.spec.ts
├── dashboard.spec.ts
├── articles.spec.ts
├── users.spec.ts
└── settings.spec.ts
```

By default Playwright creates **one worker per spec file** (up to the maximum worker limit).

```bash
Worker 1 → login.spec.ts

Worker 2 → dashboard.spec.ts

Worker 3 → articles.spec.ts

Worker 4 → users.spec.ts

Worker 5 → settings.spec.ts
```

Inside each worker:

```bash
Test 1

  ↓

Test 2

  ↓

Test 3
```

Tests inside the same spec file execute **sequentially** by default.

---

## Default Execution Flow

```bash
Spec File A
------------
Test 1
  
  ↓

Test 2

  ↓

Test 3


Spec File B
------------
Test 1
  
  ↓

Test 2


Spec File C
------------
Test 1
  
  ↓

Test 2
```

All spec files run simultaneously, but each file executes its own tests one after another.

---

## Configuration Options

### workers

Controls **how many workers** Playwright creates.

```ts
workers: undefined
```

Default behavior:

- One worker per spec file    
- Limited by CPU cores

---

Force only one worker:

```ts
workers: 1
```

Everything executes sequentially.

---

Use four workers:

```ts
workers: 4
```

Maximum of four tests/spec files execute simultaneously.

---

## fullyParallel

```ts
fullyParallel: false
```

Default:

Tests **inside a spec file** execute sequentially.

```bash
Worker

Test 1
  
  ↓

Test 2

  ↓

Test 3
```

---

When enabled:

```ts
fullyParallel: true
```

Each test inside a spec file receives its own worker.

```bash
Worker 1 → Test 1

Worker 2 → Test 2

Worker 3 → Test 3
```

Execution becomes much faster.

---

## Example Configuration

```ts
export default defineConfig({

    workers: undefined,

    fullyParallel: false,

});
```

# Controlling Parallel Execution

## Disable Parallel Execution

```ts
workers: 1
```

Everything runs one after another.

Useful when:

- debugging
- local development
- unstable environments

---

## Enable Parallel Inside One Spec

Instead of enabling it globally, you can enable it only for one file.

```ts
test.describe.configure({

    mode: "parallel",

});
```

Now every test inside this spec file runs in parallel.

---

## Parallel Only for One Describe Block

```ts
test.describe.parallel("Forms", () => {

    test(...)

    test(...)

});
```

Only tests inside this describe block execute simultaneously.

Other tests remain sequential.

---

# Serial Execution

Sometimes tests depend on each other.

Example:

```bash
Create User

	↓

Update User

	↓

Delete User
```

These cannot safely run in parallel.

Playwright supports serial execution.

```ts
test.describe.configure({

    mode: "serial",

});
```

Now:

```bash
Test A

  ↓

Test B

  ↓

Test C
```

If Test A fails:

```bash
Test B → skipped

Test C → skipped
```

This prevents dependent tests from producing misleading failures.

> **Note:** Serial execution should be used sparingly. Industry best practice is to design tests so they are independent.

---

# Renaming Spec Files to Control Execution Order

If using a single worker, Playwright executes files alphabetically.

Instead of:

```bash
login.spec.ts

dashboard.spec.ts

articles.spec.ts
```

Rename them:

```bash
001-login.spec.ts

002-dashboard.spec.ts

003-articles.spec.ts
```

Execution becomes:

```bash
001-login

	↓

002-dashboard

	↓

003-articles
```

Useful for deterministic execution when parallelism is disabled.

# Project-Specific Parallel Configuration

Parallel settings can also be configured per browser project.

Example:

```ts
projects: [

{

    name: "chromium",

    use: {

        browserName: "chromium"

    },

    fullyParallel: true

},

{

    name: "firefox",

    use: {

        browserName: "firefox"

    }

}

]
```

Chromium runs tests in parallel.

Firefox follows the global configuration.

# Best Practices

### Use parallel execution whenever possible

Independent tests benefit the most.

---

### Avoid shared test data

Bad:

```bash
Test A modifies User1

Test B modifies User1
```

Good:

```bash
Test A creates its own user

Test B creates another user
```

---

### Never depend on execution order

Tests should pass regardless of:

- execution order
- worker assignment
- browser instance

---

### Keep tests isolated

Each test should:

- create its own data
- clean up afterward
- avoid relying on previous tests

---

### Use serial mode only when necessary

Typical use cases:

- Create → Update → Delete workflows
- Multi-step migration testing
- Stateful integration tests

---

# Parallel vs Fully Parallel

|Feature|`workers`|`fullyParallel`|
|---|---|---|
|Controls number of workers|✅|❌|
|Runs spec files in parallel|✅|❌|
|Runs tests inside same spec in parallel|❌|✅|
|Default|Automatic|`false`|

---

# Key Takeaways

- A **worker** is an isolated browser process with its own clean browser context.
- By default, Playwright assigns **one worker per spec file**, and tests within each file run **sequentially**.
- Use the `workers` setting to control how many workers run simultaneously.
- Setting `fullyParallel: true` allows tests within the same spec file to execute in parallel.
- Disable parallel execution with `workers: 1` for debugging or deterministic runs.
- Use `test.describe.parallel()` or `test.describe.configure({ mode: "parallel" })` to enable parallelism for specific test groups or files.
- Use `test.describe.configure({ mode: "serial" })` only when tests have unavoidable dependencies.
- Name spec files with numeric prefixes (e.g., `001-login.spec.ts`) if execution order matters when using a single worker.
- Prefer **independent, isolated tests** to fully leverage Playwright's parallel execution capabilities in real-world automation frameworks.
