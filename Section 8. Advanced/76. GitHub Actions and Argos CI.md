# Overview

Modern QA automation is not complete without **Continuous Integration (CI)**. Instead of running Playwright tests manually on a local machine, organizations configure CI pipelines to automatically execute tests whenever code changes are pushed to a repository or a pull request is created.

In this lesson, we learn how to:

- Configure **GitHub Actions** to automatically execute Playwright tests.    
- Upload Playwright reports as build artifacts.
- Integrate **Argos CI** for visual regression testing.
- Automatically compare UI screenshots between builds.
- Approve or reject visual changes before merging code.

This workflow is widely used in production software teams to ensure both **functional correctness** and **visual consistency**.

---

# What is GitHub Actions?

GitHub Actions is GitHub's built-in CI/CD platform.

It allows you to automate workflows such as:

- Running tests    
- Building applications
- Deploying applications
- Running code quality checks
- Publishing reports
- Automating releases

A workflow is triggered by events such as:

- Push to a branch
- Pull request
- Scheduled execution
- Manual execution

Typical Playwright workflow:

```text
Developer Pushes Code
          │
          ▼
GitHub Actions Starts
          │
          ▼
Install Dependencies
          │
          ▼
Install Browsers
          │
          ▼
Run Playwright Tests
          │
          ▼
Generate Report
          │
          ▼
Upload Report Artifact
```

Create:

```text
.github/workflows/playwright.yml
```

# Basic Playwright Workflow

Example:

```yaml
name: Playwright Tests

on:
  push:
    branches:
      - main
      - master

  pull_request:
    branches:
      - main
      - master

jobs:
  test:

    runs-on: ubuntu-latest

    steps:

      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - run: npm ci --force

      - run: npx playwright install --with-deps

      - run: npm run page-objects-chrome

      - uses: actions/upload-artifact@v4
        if: always()

        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 30
```

---

# Workflow Breakdown

## Workflow Name

```yaml
name: Playwright Tests
```

This is the name displayed under GitHub Actions.

---

## Triggers

```yaml
on:
```

Defines when the workflow executes.

Example:

```yaml
push:
```

Runs after every push.

```yaml
pull_request:
```

Runs for every pull request.

---

## Job

```yaml
jobs:
```

Contains one or more jobs.

Example:

```yaml
test:
```

Creates a job named **test**.

---

## Runner

```yaml
runs-on: ubuntu-latest
```

GitHub creates a fresh Ubuntu virtual machine.

---

# Checkout Repository

```yaml
- uses: actions/checkout@v4
```

Downloads the repository into the runner.

---

# Install Node.js

```yaml
- uses: actions/setup-node@v4
```

Example:

```yaml
with:
    node-version: 20
```

Installs Node.js 20.

---

# Install Project Dependencies

```yaml
run: npm ci --force
```

`npm ci` is preferred in CI because:

- Faster than `npm install`    
- Uses `package-lock.json`
- Produces reproducible installs

---

# Install Playwright Browsers

```yaml
run: npx playwright install --with-deps
```

Installs:

- Chromium    
- Firefox
- WebKit
- Required Linux dependencies

---

# Execute Tests

Instead of running all tests:

```yaml
run: npx playwright test
```

you can execute a specific npm script:

```yaml
run: npm run page-objects-chrome
```

or a specific spec file:

```yaml
run: npx playwright test tests/login.spec.ts
```

---

# Upload Playwright Report

Even if tests fail, the report should still be available.

```yaml
if: always()
```

ensures artifacts are uploaded regardless of the test outcome.

Example:

```yaml
- uses: actions/upload-artifact@v4

  if: always()

  with:
      name: playwright-report
      path: playwright-report/
```

After the workflow finishes, the Playwright HTML report can be downloaded directly from GitHub Actions.

# What is Argos CI?

Argos CI is a visual regression testing platform.

Instead of comparing text assertions, Argos compares screenshots captured during Playwright execution.

It detects:

- Layout changes    
- Missing components
- Styling issues
- Color changes
- Font changes
- Alignment differences
- UI regressions

---

# Why Use Argos CI?

Suppose a developer changes a button color.

Functional tests still pass:

```text
Login Button Exists ✓
Button Click Works ✓
```

But visually:

```text
Blue Button

	↓

Green Button
```

Functional tests cannot detect this.

Argos CI can.

---

# How Argos Works

```text
First Build
      │
      ▼
Create Reference Screenshots
      │
      ▼
Store Baseline
      │
───────────────
Future Build
      │
      ▼
Capture New Screenshots
      │
      ▼
Compare with Baseline
      │
      ▼
Highlight Differences
```

---

# Setting Up Argos CI

Create an account on Argos using your GitHub account.

Benefits:

- Repository integration    
- Automatic pull request checks
- No manual token management (for GitHub integration)

Once connected, your GitHub repositories become available in the Argos dashboard.

---

# Install Argos Playwright Package

```bash
npm install @argos-ci/playwright --force
```

---

# Configure Playwright Reporter

Inside `playwright.config.ts`:

```typescript
reporter: [
    ['html'],
    ['@argos-ci/playwright/reporter']
]
```

This enables Argos to receive screenshots during CI execution.

---

# Configure Screenshot Behavior

Inside the `use` section:

```typescript
use: {

    trace: 'on-first-retry',

    screenshot: 'only-on-failure'
}
```

These settings are commonly used alongside Argos.

---

# Capture Screenshots in Tests

Import Argos:

```typescript
import { argosScreenshot } from '@argos-ci/playwright';
```

Capture screenshots wherever visual validation is needed:

```typescript
await argosScreenshot(page, "Form Layout Page");
```

Another example:

```typescript
await argosScreenshot(page, "Date Picker Page");
```

Each screenshot becomes part of the visual baseline.

---

# First Build

The first successful GitHub Actions run uploads screenshots to Argos.

These become the **Reference Build** (baseline).

Example:

```text
Reference Build

✓ Form Layout

✓ Date Picker
```

No comparison occurs because no previous baseline exists.

---

# Making a UI Change

Suppose a developer changes:

- Button color    
- Page layout
- Element size
- Typography

Functional tests may still pass.

Argos captures new screenshots during CI execution.

# Detecting Visual Changes

Argos reports:

```text
Changes Detected

2 Screenshots Changed
```

Each changed screenshot displays:

- Baseline image    
- Current image
- Highlighted differences
- Side-by-side comparison
- Interactive comparison slider

This allows reviewers to clearly identify visual regressions.

---

# Reviewing Visual Differences

Reviewers can inspect each screenshot and decide whether the change is:

- Expected    
- Unexpected

For example:

Baseline:

```text
Blue Submit Button
```

Current Build:

```text
Green Submit Button
```

Argos highlights only the changed region, making the review process quick and accurate.

---

# Approving Changes

If the visual modifications are intentional:

```text
Approve Changes
```

The new screenshots become the updated baseline for future comparisons.

---

# Rejecting Changes

If the changes are unintended:

```text
Reject Changes
```

The pull request remains blocked until the issue is fixed.

---

# GitHub Pull Request Integration

Argos integrates directly with GitHub pull requests.

Example status:

```text
✓ Build Passed

✗ Visual Changes Detected
```

This prevents merging until visual differences are reviewed.

Once approved:

```text
✓ Build Passed

✓ Visual Review Passed
```

The pull request can then be merged.

# Benefits of GitHub Actions + Argos CI

- Automatic Playwright execution on every push or pull request.    
- Eliminates manual test execution.
- Publishes Playwright reports automatically.
- Detects visual regressions before code is merged.
- Prevents unintended UI changes from reaching production.
- Integrates seamlessly with GitHub pull requests.
- Supports team-based review and approval workflows.

---

# Best Practices

- Keep workflow files under `.github/workflows/`.    
- Use `npm ci` instead of `npm install` in CI.
- Upload Playwright reports as artifacts.
- Capture screenshots only for important UI states.
- Review visual differences before approving.
- Maintain a clean and updated visual baseline.
- Run visual tests as part of every pull request.
