# What Is the Page Object Model?

The **Page Object Model (POM)** is a **design pattern** used in test automation to organize test code in a cleaner and more maintainable way.

Its main goals are to improve:

- **maintainability**
- **reusability**
- **readability**
- **code organization**

In a Page Object Model framework, we create a **separate class for each page** of the application (or sometimes for major reusable components of a page). That class contains the **locators** and **methods** responsible for interacting with that page.

---

# Important Note About POM

There is **no single industry-standard implementation** of Page Object Model.

The exact structure can vary depending on:

- the automation framework
- the programming language
- the project architecture
- team preferences
- the author’s style

So the Page Object Model you see in one framework may look different from another.

In this Playwright module, the idea is to follow a **Playwright-friendly POM approach**, with some additional practical improvements for writing clean and maintainable automation code.

---

# Core Concept of Page Object Model

The core concept is simple:

> **For every page of the application, create a class that contains methods for actions performed on that page.**

This helps structure the automation framework into logical sections.

For example, imagine a **Login Page** that contains:

- Email input
- Password input
- Login button

If you automate login directly inside every test, you might write something like this in multiple tests:

```typescript
await page.getByPlaceholder("Email").fill("user@email.com");
await page.getByPlaceholder("Password").fill("password123");
await page.getByRole("button", { name: "Login" }).click();
```

If several tests need to log in, you would end up repeating the same 3 lines again and again.

That leads to duplication and poor maintainability.

---

# The Problem with Repeated Test Code

Let’s say your application changes and the **Login** button text changes to **Sign In**.

If you wrote the login steps directly in many tests, you would now need to update every test file where those steps appear.

That becomes hard to maintain.

This is exactly the kind of problem that Page Object Model solves.

---

# How POM Solves It

Instead of repeating login steps in multiple tests, we move them into a reusable method inside a page object class.

For example:

```typescript
async performLogin(email: string, password: string) {
  await this.emailInput.fill(email);
  await this.passwordInput.fill(password);
  await this.loginButton.click();
}
```

Now inside the test, instead of repeating the low-level UI steps, you simply call:

```typescript
await loginPage.performLogin("user@email.com", "password123");
```

---

# Benefits of This Approach

## 1. Reduces Code Duplication

The login logic exists in **one place only**.

If the login button changes, you update the locator or method in **one class**, not in every test.

---

## 2. Improves Readability

A method name like:

```typescript
performLogin()
```

is much easier to read and understand than raw locator actions repeated in every test.

When reading the test, you can quickly understand what the test is doing at a high level.

For example:

```typescript
await loginPage.performLogin("user@email.com", "password123");
await dashboardPage.openReports();
await reportsPage.verifyMonthlyReport();
```

This reads almost like a business workflow rather than a collection of low-level UI commands.

---

## 3. Improves Maintainability

If the page structure changes, you update the page object class rather than editing many test files.

---

## 4. Improves Code Organization

Page-specific logic stays in page classes instead of being scattered across many test files.

---

# Two Important Principles to Follow

When designing a framework with Page Object Model, there are two especially important principles to keep in mind:

- **DRY** → _Don’t Repeat Yourself_    
- **KISS** → _Keep It Simple, Stupid_

---

# 1. DRY — Don’t Repeat Yourself

The **DRY** principle means you should avoid duplicating the same logic in multiple places.

If the same UI flow or logic is used repeatedly, it should probably be extracted into a reusable method or component.

### Example

Instead of writing login steps in many tests:

```typescript
await page.getByPlaceholder("Email").fill("user@email.com");
await page.getByPlaceholder("Password").fill("password123");
await page.getByRole("button", { name: "Login" }).click();
```

put them into a reusable page object method:

```typescript
await loginPage.performLogin("user@email.com", "password123");
```

---

## Practical Rule of Thumb for DRY

A useful rule is:

- If code appears **once**, it may be okay to leave it as-is.    
- If code appears **twice**, pay attention.
- If code appears **three or more times**, that’s a strong sign it should probably be extracted into a reusable function, method, or helper.

The goal is not to over-optimize every single line, but to avoid repeated logic that makes maintenance harder.

---

# 2. KISS — Keep It Simple, Stupid

The **KISS** principle means your framework should stay simple and practical.

A common mistake when people start using Page Object Model is that they become too excited and begin creating too many layers of abstraction:

- too many helper classes    
- too many utility wrappers
- too many tiny methods
- too many indirections
- overly complex framework structures

Instead of making the framework easier to maintain, this can make it harder to understand.

---

# Don’t Overengineer Your Framework

Your framework does **not** need to look “fancy” to be good.

The goal is not to impress people with complexity.

The goal is to create a framework that is:

- **reliable**    
- **easy to read**
- **easy to maintain**
- **simple enough to understand quickly**

A good automation framework should be **just structured enough** to support the project effectively—no more, no less.

---

# Useful Practice #1: Use Descriptive Naming

When designing page objects, use **clear and descriptive names** for:

- classes    
- methods
- locators
- variables
- constants
- object instances

Good naming makes the code much easier to understand.

For example:

### Better naming

```typescript
performLogin()
calendarInputField
userMenuButton
selectShippingAddress()
```

### Poor naming

```typescript
doStuff()
btn1
field
handle()
```

Descriptive names should make it obvious:

- what the method does
- what the variable stores
- what the locator points to

---

# Naming Is Hard — and Important

Naming is one of the hardest parts of programming.

If you’re unsure whether a name is clear enough:

- ask peers for feedback    
- ask for a code review
- check whether someone unfamiliar with the code would understand it quickly

Good naming has a huge impact on readability.

---

# Useful Practice #2: Avoid Tiny Methods

A common anti-pattern in Page Object Model is creating **tiny methods** that wrap only a single trivial action without adding real value.

For example:

```typescript
async clickSubmit() {
  await this.page.getByRole("button", { name: "Submit" }).click();
}
```

Technically this works, but if your framework is full of methods like this, it may become unnecessarily bloated without improving readability much.

---

# Why Tiny Methods Can Be a Problem

If every single click is wrapped into a separate one-line method, you may end up with:

- too many methods    
- harder navigation through the codebase
- extra abstraction without much benefit
- reduced clarity instead of improved clarity

The goal is not to create methods for every single click or fill action.

The goal is to create methods that represent **meaningful user actions or business flows**.

---

# Better Approach: Group Meaningful Actions

Instead of writing tiny one-line methods everywhere, prefer methods that represent a meaningful operation.

For example:

```typescript
async performLogin(email: string, password: string) {
  await this.emailInput.fill(email);
  await this.passwordInput.fill(password);
  await this.loginButton.click();
}
```

This is much more valuable than separate methods like:

```typescript
async fillEmail() {}
async fillPassword() {}
async clickLogin() {}
```

unless those smaller methods are genuinely useful across multiple workflows.

---

# What Makes a Good Page Object Method?

A good page object method usually represents:

- a meaningful user action    
- a business workflow
- a reusable page interaction
- a logical block of behavior

Examples:

- `performLogin(email, password)`
- `searchForProduct(productName)`
- `addItemToCart(productName)`
- `selectDeliveryDate(date)`
- `submitRegistrationForm(userData)`

These are much more useful than tiny methods like:

- `clickLoginButton()`
- `clickSubmitButton()`
- `typeEmail()`

unless you have a very specific reason to keep them separate.

# Main Takeaways About Page Object Model

## Page Object Model is a design pattern used to:

- organize automation code    
- reduce duplication
- improve maintainability
- improve readability
- improve reusability

---

## Core idea

- Create a separate class for each page or major page component.    
- Store page-related locators and methods inside that class.
- Call those methods from tests instead of writing raw UI steps everywhere.

---

## Benefits

- Cleaner tests    
- Reusable actions
- Centralized maintenance
- Better readability
- Better framework structure

---

## Important principles

### DRY

- Avoid repeating the same code in multiple places.    
- Extract repeated logic into reusable methods.

### KISS

- Keep the framework simple.
- Don’t overengineer or add unnecessary abstraction.

---

## Good practices

- Use descriptive names for methods, classes, and variables.    
- Prefer methods that represent meaningful user actions.
- Avoid creating too many tiny one-line methods unless they add real value.

---

# Summary

The **Page Object Model** is one of the most widely used patterns in UI test automation because it helps separate **test logic** from **page interaction logic**.

Instead of filling tests with repeated locator actions, we move page-specific behavior into reusable classes and methods. This makes tests easier to read, easier to update, and easier to scale as the framework grows.

The most important thing is to use POM **practically**:

- reduce duplication    
- keep code readable
- avoid overengineering
- keep the framework simple and maintainable
