In this lesson, we’ll continue refactoring our **Navigation Page** and make changes related to **locators**.

According to Playwright’s recommendation for Page Objects, locators should be **separated from functional methods**.

For example, this is a functional method:

```typescript
async formLayoutsPage() {
  await this.selectGroupMenuItem("Forms");
  await this.page.getByText("Form Layouts").click();
}
```

Here, the locator (`getByText("Form Layouts")`) is defined directly inside the method.  
However, Playwright recommends keeping all locators separately from the methods themselves.

---

## Separating Locators from Methods

Let’s see how to do that.

First, we create **class fields** for all locators we want to remove from the methods:

- Form Layouts
- Datepicker
- Smart Table
- Toastr
- Tooltip

### Step 1: Add locator fields

```typescript
readonly formLayoutMenuItem: Locator;
readonly datePickerMenuItem: Locator;
readonly smartTableMenuItem: Locator;
readonly toastrMenuItem: Locator;
readonly tooltipMenuItem: Locator;
```

Each field has a meaningful name and uses the `Locator` type imported from Playwright.

---

### Step 2: Initialize locators in the constructor

Now we assign actual locator values to these fields inside the constructor:

```typescript
constructor(page: Page) {
  this.page = page;
  this.formLayoutMenuItem = page.getByText("Form Layouts");
  this.datePickerMenuItem = page.getByText("Datepicker");
  this.smartTableMenuItem = page.getByText("Smart Table");
  this.toastrMenuItem = page.getByText("Toastr");
  this.tooltipMenuItem = page.getByText("Tooltip");
}
```

---

### Step 3: Use locator fields inside methods

Now the methods become cleaner because they use the pre-defined locator fields:

```typescript
async formLayoutsPage() {
  await this.selectGroupMenuItem("Forms");
  await this.formLayoutMenuItem.click();
}
```

Instead of calling `page.getByText("Form Layouts")` directly inside the method, we use `this.formLayoutMenuItem`.

The same pattern can be applied to the rest of the methods.

---

## Refactored `navigationPage.ts`

```typescript
import { Locator, Page } from "@playwright/test";

export class NavigationPage {
  readonly page: Page;
  readonly formLayoutMenuItem: Locator;
  readonly datePickerMenuItem: Locator;
  readonly smartTableMenuItem: Locator;
  readonly toastrMenuItem: Locator;
  readonly tooltipMenuItem: Locator;

  constructor(page: Page) {
    this.page = page;
    this.formLayoutMenuItem = page.getByText("Form Layouts");
    this.datePickerMenuItem = page.getByText("Datepicker");
    this.smartTableMenuItem = page.getByText("Smart Table");
    this.toastrMenuItem = page.getByText("Toastr");
    this.tooltipMenuItem = page.getByText("Tooltip");
  }

  async formLayoutsPage() {
    await this.selectGroupMenuItem("Forms");
    await this.formLayoutMenuItem.click();
  }

  async datePickerPage() {
    await this.selectGroupMenuItem("Forms");
    await this.page.waitForTimeout(1000);
    await this.datePickerMenuItem.click();
  }

  async smartTablePage() {
    await this.selectGroupMenuItem("Tables & Data");
    await this.smartTableMenuItem.click();
  }

  async toastrPage() {
    await this.selectGroupMenuItem("Modal & Overlays");
    await this.toastrMenuItem.click();
  }

  async tooltipPage() {
    await this.selectGroupMenuItem("Modal & Overlays");
    await this.tooltipMenuItem.click();
  }

  private async selectGroupMenuItem(groupItemTitle: string) {
    const groupMenuItem = this.page.getByTitle(groupItemTitle);
    const expandedState = await groupMenuItem.getAttribute("aria-expanded");

    if (expandedState === "false") {
      await groupMenuItem.click();
    }
  }
}
```

# Why Playwright recommends this approach

The main idea is **reusability**.

If the same locator is needed in multiple methods, you define it once and reuse it everywhere. That helps avoid duplication and follows the **DRY principle** (**Don’t Repeat Yourself**).

For example, if multiple methods need to interact with the same element, having a dedicated locator field prevents you from repeating the same locator string over and over.

---

# Potential downsides of this approach

The instructor points out that, in real projects, this approach can make page objects **too large and harder to read**.

Imagine a page object with:

- 15–20 methods
- multiple locators per method

Over time, the class may end up with **dozens of locator fields** at the top and **dozens of assignments** in the constructor.

That can lead to a few problems:

---

## 1. Too many locator fields

A page object can become cluttered with a long list of locators:

```typescript
readonly formLayoutMenuItem: Locator;
readonly datePickerMenuItem: Locator;
readonly smartTableMenuItem: Locator;
readonly toastrMenuItem: Locator;
readonly tooltipMenuItem: Locator;
// ... many more
```

And the constructor also grows with many locator assignments.

This can make the file harder to scan and maintain.

---

## 2. Duplicate locators with different names

When multiple engineers work on the same project, it’s easy to accidentally create duplicate locators under different names.

For example:

```typescript
readonly tooltipMenuItem: Locator;
readonly tooltipMenuSelector: Locator;
```

Both might point to the exact same element, just with different naming.

---

## 3. Dead locators

Sometimes locators stay in the class even though no method actually uses them anymore.  
This creates unused code that makes the page object noisy.

---

## 4. Some locators can’t be cleanly extracted

Take this helper method:

```typescript
private async selectGroupMenuItem(groupItemTitle: string) {
  const groupMenuItem = this.page.getByTitle(groupItemTitle);
  const expandedState = await groupMenuItem.getAttribute("aria-expanded");

  if (expandedState === "false") {
    await groupMenuItem.click();
  }
}
```

Here, the locator depends on a **method parameter** (`groupItemTitle`), so it doesn’t fit nicely into a fixed constructor-based locator field.

That means you may still end up with some locators inside methods anyway, resulting in a mixed style.

---

# Instructor’s preferred approach

The instructor personally prefers **keeping locators inside the functional methods** instead of moving them all into the constructor.

### Why?

Because it keeps the code simpler and easier to debug.

For example, when a test fails, you can immediately open the method and see the exact locator used on that failing line.

Example:

```typescript
async tooltipPage() {
  await this.selectGroupMenuItem("Modal & Overlays");
  await this.page.getByText("Tooltip").click();
}
```

This is very direct:

- you see the action,
- you see the locator,
- and debugging is straightforward.

---

# Instructor’s recommendation

The instructor prefers following **KISS**:

> **Keep It Simple**

So instead of over-engineering the page object with many locator fields, they keep locators inside methods unless there’s a strong reason to extract them.

According to the instructor:

- it’s easier to read,
- easier to debug,
- easier to maintain,
- and duplication usually stays very low if methods are designed well.

---

# Key takeaway from this lesson

There are **two common styles** for handling locators in Page Objects:

---

## Style 1: Separate locators into class fields / constructor

### Pros

- Reusable locators
- Follows DRY
- Clear separation between locators and actions

### Cons

- Can make page objects large and cluttered
- Easy to create duplicate or unused locators
- Some dynamic locators still need to stay inside methods

---

## Style 2: Keep locators inside functional methods

### Pros

- Simpler and more readable
- Easier to debug
- Less boilerplate

### Cons

- Some locator duplication may happen
- Less centralized than the constructor-based style

---

# Final note

Playwright **recommends separating locators from methods**, but there is no single “perfect” way to build Page Objects.

The instructor’s personal preference is to **keep locators inside the methods** and follow the **KISS principle**, because in practice it often leads to cleaner and easier-to-maintain test code.
