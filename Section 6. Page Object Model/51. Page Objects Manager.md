In this lesson, we’ll continue improving our framework and refactor it a bit more to better organize the code.

By now, you’ve probably noticed a pattern building up in our tests:

- we import page objects into the spec file    
- we create instances of those page objects inside each test
- we repeat the same setup again and again for every test

That works fine for a small demo project, but imagine a real project with **20 pages** in the application. That would mean:

- **20 page object imports**
- **20 page object instances**
- repeating that setup in test after test

That’s not a very scalable approach.

So in this lesson, we’ll create a **Page Object Manager** (sometimes called a Page Manager). Its job is to **centralize access to all page objects** so that each test only needs **one manager instance**, and the manager will give us access to the page objects we need.

---

# Why use a Page Object Manager?

Without a manager, a test can start looking like this:

```typescript
const navigateTo = new NavigationPage(page);
const onFormLayoutsPage = new FormLayoutPage(page);
const onDatePickerPage = new DatePickerPage(page);
```

That’s manageable with 2–3 page objects, but it becomes repetitive when your framework grows.

A **Page Object Manager** solves that by:

- creating all page object instances in one place    
- exposing methods to access them
- reducing duplication in test files
- improving readability

---

# Create the `PageManager`

We’ll create a new file inside the `pages` folder called:

```text
pageManager.ts
```

## `pageManager.ts`

```typescript
import { Page } from "@playwright/test";
import { NavigationPage } from "./navigationPage";
import { FormLayoutPage } from "./formLayoutsPage";
import { DatePickerPage } from "./datePickerPage";

export class PageManager {
  private readonly page: Page;
  private readonly navigationPage: NavigationPage;
  private readonly formLayoutPage: FormLayoutPage;
  private readonly datePickerPage: DatePickerPage;

  constructor(page: Page) {
    this.page = page;
    this.navigationPage = new NavigationPage(this.page);
    this.formLayoutPage = new FormLayoutPage(this.page);
    this.datePickerPage = new DatePickerPage(this.page);
  }

  navigateTo() {
    return this.navigationPage;
  }

  onFormLayoutsPage() {
    return this.formLayoutPage;
  }

  onDatePickerPage() {
    return this.datePickerPage;
  }
}
```

---

# How the `PageManager` works

Let’s break down what’s happening here.

## 1. Store the shared Playwright `page`

```typescript
private readonly page: Page;
```

We keep a reference to the Playwright `page` fixture so that all page objects use the **same browser page instance**.

Inside the constructor:

```typescript
this.page = page;
```

---

## 2. Create fields for each page object

```typescript
private readonly navigationPage: NavigationPage;
private readonly formLayoutPage: FormLayoutPage;
private readonly datePickerPage: DatePickerPage;
```

These fields will hold instances of our page objects.

---

## 3. Initialize all page objects inside the constructor

```typescript
this.navigationPage = new NavigationPage(this.page);
this.formLayoutPage = new FormLayoutPage(this.page);
this.datePickerPage = new DatePickerPage(this.page);
```

Each page object gets the **same Playwright page instance**, which keeps everything consistent during test execution.

---

## 4. Expose methods that return page objects

```typescript
navigateTo() {
  return this.navigationPage;
}

onFormLayoutsPage() {
  return this.formLayoutPage;
}

onDatePickerPage() {
  return this.datePickerPage;
}
```

These methods act like access points to your page objects.

This lets you write test code like:

```typescript
pm.navigateTo().formLayoutsPage();
pm.onFormLayoutsPage().submitInlineFormWithNameEmailAndCheckbox(...);
pm.onDatePickerPage().selectDatePickerWithRangeFromToday(...);
```

---

# Refactor the spec file to use the Page Manager

Now let’s update the spec file so that instead of creating multiple page object instances directly, we create **just one `PageManager` instance**.

## `pageObjectModel.spec.ts`

```typescript
import { test, expect } from "@playwright/test";
import { PageManager } from "../pages/pageManager";

test.beforeEach(async ({ page }) => {
  await page.goto("http://localhost:4200");
});

test("First Page Object", async ({ page }) => {
  const pm = new PageManager(page);

  await pm.navigateTo().formLayoutsPage();
  await pm.navigateTo().datePickerPage();
  await pm.navigateTo().smartTablePage();
  await pm.navigateTo().toastrPage();
  await pm.navigateTo().tooltipPage();
});

test("Parametrized Methods", async ({ page }) => {
  const pm = new PageManager(page);

  await pm.navigateTo().formLayoutsPage();
  await pm
    .onFormLayoutsPage()
    .submitUsingTheGridFormWithCredentialsAndSelectOption(
      "test@test.com",
      "Welcome1",
      "Option 2"
    );

  await pm
    .onFormLayoutsPage()
    .submitInlineFormWithNameEmailAndCheckbox(
      "John Smith",
      "john@test.com",
      true
    );
});

test("Date Picker Page Object", async ({ page }) => {
  const pm = new PageManager(page);

  await pm.navigateTo().datePickerPage();
  await pm.onDatePickerPage().selectCommonDatePickerDateFromToday(5);
  await pm.onDatePickerPage().selectDatePickerWithRangeFromToday(6, 15);
});
```

---

# What improved after this refactor?

## Before

Each test needed to create page objects manually:

```typescript
const navigateTo = new NavigationPage(page);
const onFormLayoutsPage = new FormLayoutPage(page);
const onDatePickerPage = new DatePickerPage(page);
```

## After

Each test creates only one object:

```typescript
const pm = new PageManager(page);
```

Then you access page objects through it:

```typescript
await pm.navigateTo().formLayoutsPage();
await pm.onFormLayoutsPage().submitInlineFormWithNameEmailAndCheckbox(...);
await pm.onDatePickerPage().selectCommonDatePickerDateFromToday(5);
```

---

# Benefits of the Page Manager pattern

## 1. Less duplication

You no longer repeat page object instantiation in every test.

---

## 2. Cleaner test files

Your tests focus more on **test flow** and less on **setup noise**.

---

## 3. Better readability

This reads almost like plain English:

```typescript
await pm.navigateTo().datePickerPage();
await pm.onDatePickerPage().selectCommonDatePickerDateFromToday(5);
```

That makes it much easier to understand what the test is doing.

---

## 4. Easier scaling

As your framework grows, you just add new page objects inside the `PageManager` constructor and expose them through methods.

For example, if you later create:

- `SmartTablePage`    
- `ToastrPage`
- `LoginPage`

you can register them in one place and use them everywhere through the same manager instance.

---

# Recommended naming style

The naming used here is intentional:

- `navigateTo()` → returns the navigation page object    
- `onFormLayoutsPage()` → returns the form layouts page object
- `onDatePickerPage()` → returns the date picker page object

This makes the test code read naturally:

```typescript
await pm.navigateTo().datePickerPage();
await pm.onDatePickerPage().selectDatePickerWithRangeFromToday(6, 15);
```

That improves both **readability** and **maintainability**.

# Final structure after this lesson

At this point, your framework likely looks something like this:

```text
pages/
├── navigationPage.ts
├── formLayoutsPage.ts
├── datePickerPage.ts
└── pageManager.ts

tests/
└── pageObjectModel.spec.ts
```

This is a much more maintainable structure than managing all page object instances directly inside each test.

# Summary

In this lesson, we created a **Page Object Manager** to reduce duplication and better organize our framework.

## What we did

- created a new `PageManager` class    
- stored all page object instances inside it
- initialized those page objects in one place
- exposed helper methods to access each page object
- updated tests to use **one manager instance** instead of multiple page object instances

## Main advantage

Instead of this in every test:

```typescript
const navigateTo = new NavigationPage(page);
const onFormLayoutsPage = new FormLayoutPage(page);
const onDatePickerPage = new DatePickerPage(page);
```

you now do this:

```typescript
const pm = new PageManager(page);
```

That makes your tests:

- shorter    
- cleaner
- easier to scale
- easier to read
