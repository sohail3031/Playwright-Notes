# ✅ Test Scenarios — Successful Login

## TC-LOGIN-001 — Valid login with correct credentials

- Navigate to `/auth/login`
- Enter `customer@practicesoftwaretesting.com / welcome01`
- Click **"Login"**

**Expected:**

- User is redirected to the account/dashboard page
- A **"My Account"** link or logout option appears in the nav bar
- Login form is no longer visible

---

## TC-LOGIN-002 — Logged-in user sees personalized greeting

- Log in as **Jane Doe**    

**Expected:**

- A welcome message or first name **"Jane"** appears somewhere in the account area

---

## TC-LOGIN-003 — Admin login leads to a valid session

- Log in with `admin@practicesoftwaretesting.com / welcome01`    

**Expected:**

- Login succeeds _(admin uses the same login flow as customers on the front-end)_

---

# ❌ Test Scenarios — Failed Login

## TC-LOGIN-004 — Login fails with incorrect password

- Enter `customer@practicesoftwaretesting.com / wrongpassword123`    

**Expected:**

- An error message is displayed _(e.g. "Invalid credentials")_
- User remains on the login page
- No session/auth token is created _(nav bar still shows "Login")_

---

## TC-LOGIN-005 — Login fails with unregistered email

- Enter `doesnotexist@nowhere.com / anyPassword1`    

**Expected:**

- Same generic error message is shown _(no information leak about which field was wrong)_

---

## TC-LOGIN-006 — Login fails with empty email field

- Leave email blank, enter password only, click **Login**    

**Expected:**

- Inline validation message appears under the email field _(e.g. "Email is required")_
- Form is not submitted

---

## TC-LOGIN-007 — Login fails with empty password field

- Enter valid email, leave password blank, click **Login**    

**Expected:**

- Inline validation message appears under the password field
- Form is not submitted

---

## TC-LOGIN-008 — Login fails with both fields empty

- Click **Login** with both fields blank    

**Expected:**

- Both field-level validation messages appear simultaneously

---

## TC-LOGIN-009 — Login fails with malformed email format

- Enter `notanemail` _(no `@` symbol)_ as email    

**Expected:**

- Validation message indicates invalid email format

---

# 🔒 Test Scenarios — Session & Navigation

## TC-LOGIN-010 — Logout clears the session

- Log in successfully, then click **"Logout"**    

**Expected:**

- User is redirected to homepage or login page
- Nav bar reverts to showing **"Login" / "Register"** links
- Attempting to revisit the account page redirects back to login

---

## TC-LOGIN-011 — Accessing account page while not logged in redirects to login

- Without logging in, navigate directly to `/account`    

**Expected:**

- User is redirected to `/auth/login`

---

## TC-LOGIN-012 — "Register" link navigates to the registration page

- From the login page, click **"Register" / "Create an account"**    

**Expected:**

- URL changes to `/auth/register`
- Registration form is displayed

# 🔧 Locator Practice Targets

- `getByPlaceholder('Your email')`    
- `getByPlaceholder('Your password')`
- `getByRole('button', { name: 'Login' })`
- `getByText('Invalid')` for error message assertions
- Parent/child: locate the error message within a specific form field's container, not just anywhere on page
